# COMPREHENSIVE PLAN FOR THE DESIGN OF FRANKENLEAN

**Working name:** FrankenLean (`franken_lean`, crate prefix `fln-`)
**Native sovereignty (the Oracle-Only Law):** the entire toolchain — parser, macro engine, elaborator, unifier, instance engine, tactic framework, simp and the decision procedures, kernel, compiler, runtime, build system, and server — is **native Rust, end to end**. No upstream implementation code executes as a component of FrankenLean, ever: not the C++ kernel, not the self-hosted `Lean.*` elaborator sources, not stage0. The Reference toolchain and its sources appear in exactly one place — inside the Tribunal, as the differential oracle, fixture generator, and census-extraction source. The only Lean code FrankenLean ever *executes* is user code (mathlib's tactics, downstream libraries, lakefiles, `#eval`), because executing user metaprograms is what the Lean language *is*; the engine that runs them (Golem) and every builtin they call (the Mirror, §4.3) are ours
**Language:** Rust, edition 2024, on an exact pinned nightly toolchain recorded in `SUITE.lock` (no "or later")
**Safety target:** no project-authored `unsafe` anywhere outside the explicitly isolated boundary crates (`fln-unsafe-abi`, `fln-unsafe-region`, `fln-unsafe-jit`); `#![forbid(unsafe_code)]` in every authoritative fln crate; the full transitive dependency closure is pinned, allowlisted, and audited for `unsafe` and native-code exposure
**Dependency target:** `std` plus the Dicklesworthstone-owned FrankenSuite — `asupersync`, `frankensqlite`, `franken_networkx`, `frankensearch`, `frankentui`, `franken_markdown` (with its `fmd-font`/`fmd-math` workspace crates), `fastmcp_rust`, `atp` — under the governed-closure doctrine of §3. Optional tier, feature-gated and never on the critical path: `frankentorch` (learned ranking), `franken_node` (widget JS host)
**External tools policy:** **zero mandatory external tools.** Two optional tools are permitted, both inherited from the Reference's own contract and both eliminable: a system C compiler, used *only* by the `--backend c` compatibility code generator (exactly as upstream `leanc` requires one; the native Iron backend needs none), and system `git`, used *only* by Lake-compatible dependency fetching (exactly as upstream Lake shells out to it; the Ledger's CAS transport rides `atp` instead wherever both sides speak it). There is no GMP, no libuv, no CaDiCaL, no C++ runtime, no vendored C of any kind on any path, ever
**Compatibility contract:** FrankenLean is a **drop-in replacement at four binary surfaces** — the source language, the `.olean` object format (read *and* write), the `lean_object` C ABI (`lean.h` twin, so precompiled native plugins and `@[extern]` code interoperate), and the LSP wire protocol including the `$/lean/*` extensions — plus the `.ilean` editor-metadata files, the generated-C/object artifact chain, and the CLI/exit-code/`--json` surfaces of `lean`, `leanc`, and `lake`. Under those familiar surfaces it is deliberately *better where better is sound*: deterministic under parallelism, declaration-granular incremental, memory-shared, provenance-transparent. Every intentional divergence is documented in the user-facing **Behavior Notes** (§18.8) and gated by the mode system of §4
**System class:** a complete interactive theorem prover and dependently-typed programming language implementation, native Rust throughout: parser/macro engine, elaborator, trusted kernel for the Lean 4 type theory (dual-engine with an independent checker and consensus receipts), metaprogram compiler and virtual machine executing the existing Lean tactic ecosystem unmodified against a natively-implemented `Lean.*` API surface, runtime and ABI twin, owned bignum and owned deterministic float transcendentals, module codec, content-addressed build fabric, always-on language server, rewrite/decision engines, and an agent-native service surface — usable as a toolchain (`lean`/`lake` drop-ins), an embeddable Rust library, and an MCP server
**Reference:** `leanprover/lean4` @ tag `v4.32.0` (stable, 2026-07-13; the pinned commit hash is recorded in `SUITE.lock` beside the tag) — pinned as **the Reference**: the immutable *oracle* of the language semantics, the type theory, the object formats, the wire protocols, the builtin census, and the conformance fixtures. The Reference is a semantics oracle and a compatibility target, never a runtime component: release binaries contain no code path that invokes it (§3 D8). Toolchain releases through `v4.33.0-rc1` (2026-07-15) were surveyed for trajectory; the pin advances only by the epoch policy of §22.5. `leanprover-community/mathlib4` at its `v4.32.0`-compatible commit is pinned as **the Corpus**: the de-facto behavioral specification of everything the Reference's documentation does not say
**Document date:** 2026-07-20
**Document status:** bold greenfield architecture and execution program; intentionally not an MVP specification. **Revision 1.** This plan follows the house structure of the FrankenGraphDB and FrankenManim plans: a Reference anatomy that is a semantic audit rather than an homage, a foundation audit at the symbol level, a constitutional constraints section, named subsystems specified at full strength, a conformance Gauntlet with typed claims, CI-enforced performance gates derived from measured baselines, and workstreams sequenced by dependency rather than by scope reduction.

## 0. Declaration of intent

FrankenLean should not be designed as "Lean with Rust syntax," "a kernel checker with ambitions," or a transpiler that shells out to the real `lean` when things get hard. Those paths would produce competent software and miss the opportunity.

The opportunity: Lean 4 — Leonardo de Moura's prover-language, now stewarded by the Lean FRO — is one of the most important software artifacts of the decade. Its *ideas* are superb: a small trusted kernel over a rich dependent type theory, an elaborator written in the language itself, hygienic user-extensible syntax all the way down, a metaprogramming model where `simp` and `grind` are just library code, and a 1.5-million-line mathematical corpus (mathlib: over 115,000 definitions and 232,000 theorems) that no other prover ecosystem can match. Its *substrate* is an accident of history: a C++ kernel and runtime nobody has verified, a GMP dependency for every literal, a vendored libuv event loop, an external CaDiCaL binary behind `bv_decide`, a mandatory C compiler behind every executable, a bootstrap whose stage0 is megabytes of checked-in generated C updated in nearly every nightly, a pointer-dump object format, a process-per-file server that pays ~60 seconds and gigabytes of RSS to import mathlib *per worker*, file-granular invalidation that turns a one-line leaf edit into an afternoon, and — as of the current toolchains — an async elaboration mode that is documented in the community's own bug reports to *change elaboration results depending on the schedule*.

Every one of those is a substrate problem, not an idea problem. The FrankenSuite exists precisely to replace substrates: it has already rebuilt the database (frankensqlite), the async operating layer with deterministic simulation and formal anchors (asupersync), the graph brain (franken_networkx), the search stack (frankensearch), the terminal (frankentui), the document/typesetting stack (franken_markdown + fmd-math), the agent protocol layer (fastmcp_rust), and the bulk-transfer plane (atp). The missing piece is not any of those. It is the prover itself — the one system class where the suite's obsessions (determinism as a contract, content addressing, capability contexts, claims with types, verification of the implementation, sovereignty of the dependency closure) are not merely virtues but *the entire point of the product category*. A theorem prover whose own implementation cannot be replayed, audited, or independently re-checked is an irony the field has simply learned to live with. We decline to.

The design commitments:

1. **Drop-in at the binary surfaces, radical underneath.** FrankenLean reads and writes the Reference's `.olean` modules bit-compatibly, exports the `lean_*` C ABI so existing native plugins load, speaks the Reference's LSP dialect so `vscode-lean4` connects unmodified, accepts `lakefile.lean`/`lakefile.toml`/`lean-toolchain` as-is, and elaborates the Corpus to statement-identical environments. Interop is bidirectional and continuous: a project may mix FrankenLean-produced and Reference-produced artifacts mid-build, and the Tribunal (§18) proves it every night. Everything behind those surfaces — the scheduler, the store, the server topology, the incrementality granule, the diagnostics — is rebuilt without sentiment.
2. **No MVP.** Every subsystem is specified at full strength. Sequencing is by dependency, not scope reduction; the gates in §22 are integration checkpoints, and G0 exists so no later gate begins with its load-bearing unknowns unresolved.
3. **The bootstrap is dead.** The Reference compiles itself: its elaborator and compiler are Lean programs, bootstrapped through a checked-in stage0 of generated C, so every deep change pays the stage-update treadmill and every reader of the system confronts a Möbius strip. FrankenLean's host — parser, elaborator, kernel, compiler, VM, server — is native Rust, buildable by `cargo` from a cold checkout with no prior Lean in the universe. Lean-language *user* code still runs everywhere it should (that is commitment 4); upstream implementation code never runs at all, and nothing Lean-shaped ever sits underneath the machine that runs it.
4. **Native machine, native Mirror — user code unmodified.** The tactic ecosystem is Lean code calling the `Lean.*` API. FrankenLean therefore implements that entire API surface **natively** behind a census-generated façade and executes *unmodified user metaprograms* — mathlib's tactics and everything downstream — on its own VM (Golem, §11) against the same ABI-valued state the native machinery manipulates. The upstream implementation never executes as a component; its sole runtime role is as the Tribunal's differential oracle (the Oracle-Only Law, D8). This is the architectural crux of the whole program, stated as Bet B2 in §0.1 and specified in §4.3: it is what makes "mathlib's tactics run unmodified on a fully native toolchain" an engineering plan rather than a slogan.
5. **Determinism as a contract, not a hope.** Same input closure ⇒ same environment, same diagnostics, same artifacts, at any thread count, on any schedule — an invariant (FL-INV class), enforced by construction through asupersync's deterministic scheduling discipline and the CGSE doctrine franken_networkx already ships for graph algorithms, and tested at {1, 8, 32} threads on every commit. The Reference's schedule-sensitive elaboration results are the negative example this contract exists to delete.
6. **The kernel answers to no one — and shows receipts.** Crucible (§8) is a dual-engine trusted checker with a hard size covenant, deterministic fuel parity with the Reference, proof-certificate export by default, and a standing differential harness against the Reference kernel, lean4lean, and lean4checker. Independence of the checking path is a product feature: FrankenLean is, from day one, the strongest independent verifier of the existing Lean world, before it is anything else.
7. **Contracts before construction.** Gate G0 (§22.1) retires the load-bearing unknowns — the ABI resurrection of a real mathlib `.olean`, the byte-identical re-emit, Golem execution of real user-class code, defeq tail latency, hygiene fidelity, heartbeat parity, the Mirror-façade viability and instrumented-oracle spikes, the dependency-closure audit — as compile-tested spikes before W2–W12 freeze interfaces.

## 0.1 The thesis: eight bets

The leapfrog is not one trick. It is the *composition* of eight bets, each individually at or beyond the current frontier, made feasible only because the foundation libraries already exist:

| # | Bet | One-line statement | Why nobody else can do it |
|---|-----|--------------------|---------------------------|
| **B1** | **The Ledger** | The environment is a Merkle DAG of content-addressed declarations; builds are memoized queries over it; a one-line leaf edit re-elaborates its true dependency cone (seconds), not its file cone (hours); the "cloud cache" is native CAS sync over atp, not a bolted-on download script. | Requires a durable content-addressed store (frankensqlite), a deterministic elaborator whose outputs are cacheable by hash (B5), and fountain-coded transfer (atp) designed in from byte zero. |
| **B2** | **The Native Mirror** | The entire `Lean.*`/`Init`/`Std` builtin surface that metaprograms and compiled code can reach is served *natively*: toolchain-API symbols are Rust implementations registered under upstream names behind a census-generated façade; pure library code is upstream-authored *source* elaborated by our own toolchain; user metaprograms (mathlib's tactics and below) run on our VM against that native surface and cannot tell. The Reference never executes as a component — it is the Tribunal's oracle. | Requires owning the `lean_object` ABI (Marrow), the full `.olean` codec (Grimoire), a native elaborator whose fidelity is proven differentially at mathlib scale (Athanor + Tribunal), and a VM (Golem) simultaneously; every prior reimplementation stopped at the kernel because it lacked all four — and every hosted-frontend design forfeits the point of a reimplementation. |
| **B3** | **Kernel with receipts** | A ≤ 12 KLOC dual-engine trusted checker (certified small-step + NbE accelerator, cross-checked), deterministic fuel parity, compressed proof-certificate export by default, consensus receipts with an in-repo independent checker plus external witnesses (lean4checker/lean4lean via the export format), and CI-grade differential checking against foreign kernels — disagreement halts, never outvotes. | Requires the suite's claims-have-types discipline and asupersync's formal-anchor posture; upstream cannot ship this without breaking its own TCB story mid-flight. |
| **B4** | **Deterministic parallel elaboration** | Declaration-granular dataflow parallelism with speculative execution and deterministic merge: results are schedule-independent by construction, so parallelism is free to be aggressive. | Requires asupersync's lab-replayable scheduler and the CGSE tie-break doctrine; upstream's `Elab.async` demonstrably lacks the invariant. |
| **B5** | **Rewriting at machine speed** | simp-compatible rewriting backed by compiled discrimination automata shipped as per-library indexes, an e-graph saturation lane with kernel-checked proof extraction, and owned decision procedures (Verdict CDCL with owned proof checking) replacing the external-solver TCB. | Requires the freedom to add engines *under* a fixed semantics (kernel-checked outputs make every engine untrusted) and the sovereignty rule that forbids shelling to CaDiCaL. |
| **B6** | **Agent-native by construction** | The prover as an agent-legible service: an MCP surface (fastmcp_rust), semantic library search (frankensearch), structured machine-readable proof states, O(1) proof-state snapshots for search trees, and replayable elaboration traces — purpose-built for the 2026 reality in which frontier labs benchmark their reasoning models *inside Lean*. | Requires the daemon architecture (B1's shared heap kills the ≈60 s/branch import tax that today's DSP pipelines document), Palimpsest's provenance, and a protocol stack the suite already owns. |
| **B7** | **The causal proof graph** | Every declaration, instance selection, simp firing, macro expansion, capability use, kernel verdict, and build product is a node in a typed provenance graph with edge families (`READS_TYPE` vs `READS_VALUE` vs `UNFOLDS`, `SELECTS_INSTANCE`/`REJECTS_INSTANCE`, ...) and *completeness classes*; impact cones, semantic blame slices, semantic diff, fragility signals, and conflict-aware **semantic merge** become queries — and only Complete/Conservative edges may drive cache invalidation, welding provenance to soundness. | Requires Palimpsest's always-on journaling, the Ledger's content identities, and franken_networkx's deterministic graph machinery to already exist; upstream has no journal to project a graph from. |
| **B8** | **Evidence-native engineering** | Every public claim — compatibility, soundness, determinism, speed, recovery — is a row in a machine-checked claim matrix with an evidence state (OBSERVED/TARGETED/HYPOTHESIS/PROVEN/BLOCKED), a freshness bound, and a reproduction command; documentation CI rejects wording stronger than the matrix permits; compatibility is reported per-surface at evidence levels L0-L4 and per-release at R0-R5, never as one percentage. | Requires the claims-have-types discipline (D7) and the Tribunal's generated ledgers; a project without executable censuses can only assert. |

The 2026 market context makes the timing exact: Lean has become the venue where frontier labs argue about machine reasoning — dedicated proof models ship monthly, DSP-style parallel tactic search is a publishing subfield, and the published bottlenecks of that entire research program are *implementation* facts of the Reference: ~60 seconds of mathlib import per search branch, 18–735 seconds of re-elaboration per theorem context, one process per file, no sub-declaration state sharing. Meanwhile the mathematical community's own bottleneck is the mathlib build treadmill and the monthly toolchain migration. Both communities are blocked on the substrate. FrankenLean is the substrate.

**Anti-goals of this document:** There is no "MVP", no "v1 that only checks kernels", no "phase where we shell out to real `lean` temporarily", and — constitutionally — no phase where upstream's *implementation* runs as a component of ours: no hosted C++ kernel, no `Lean.Elab` sources on our VM standing in for an elaborator we haven't written, no "compatibility via running their frontend." The real `lean` appears exactly once in this program: inside the Tribunal, as the differential oracle and fixture mine it deserves to be. (During construction, a development-only lockstep harness may run the Reference beside us for diagnosis; any result it touches is marked `ORACLE_FALLBACK`, can never satisfy a gate or enter an attested cache, and the harness is compiled out of release binaries — §18.10.) Changing the type theory is out of scope forever: FrankenLean proves the *same theorems* under the *same axioms* — `propext`, `Quot.sound`, `Classical.choice` — with the same kernel judgments, or it is not Lean. Language design is out of scope for 1.0: new surface syntax, new tactics, and new elaboration behaviors ship only behind the `frontier` mode of §4 and never leak into `faithful` or `sound` artifacts.

## 1. Anatomy of the Reference: what Lean 4 actually is

The pinned tag is a C++/Lean hybrid: a C++ kernel and runtime (order tens of KLOC), a compiler/elaborator/server written in Lean itself (order hundreds of KLOC under `src/Lean`), a checked-in stage0 of generated C (order millions of lines) that bootstraps it, Lake and the standard library in Lean, and a vendored libuv. Every claim below is to be re-verified against the pin during W1's Reference audit and frozen into the conformance fixture set; this section reads as the **semantic specification and design reference** — the things users' code, libraries, and mental models depend on — plus an honest record of mechanisms we deliberately do *not* copy (marked ↷, with the replacement noted).

### 1.1 The term plane

Kernel expressions are a small closed inventory: bound variables (de Bruijn indices), free variables, metavariables (elaboration-only; never seen by the kernel), sorts, constants applied to universe-level lists, applications, lambdas, dependent function types, `let`, literals (`Nat` and `String`), metadata wrappers, and structure projections. Universe levels form their own tiny language: `zero`, `succ`, `max`, `imax` (with the load-bearing `imax u 0 = 0` collapse that makes `Prop` impredicative), parameters, and level metavariables. `Prop = Sort 0`, `Type u = Sort (u+1)`. Expressions are hash-consed in practice and carry cached hashes, loose-bvar ranges, and has-fvar/has-mvar flags used pervasively for fast traversal pruning. All of this is API surface — metaprograms pattern-match on it — and FrankenLean preserves the constructor inventory, the flag semantics, and the hashing observables exactly (§6, §8).

### 1.2 The kernel plane

The trusted plane, measured at the pin (approximate source weight, to fix intuitions): `type_checker.cpp/h` ≈56 KB (infer/defeq/lazy-whnf/caches), `inductive.cpp/h` ≈65 KB (positivity, recursor generation, nested/mutual), the `expr`/`level`/`declaration`/`environment` families ≈140 KB, plus term surgery, quotients, and the equiv manager — **≈315 KB of C++ before counting the runtime objects it manipulates, GMP beneath its literals, and platform libm beneath `Float`**. One engine, one judgment, no receipt, no independent witness, no promised determinism across platforms (pointer-keyed caches, hash-iteration order, libm drift). This census is why Crucible's covenant (§8.1), the consensus posture (§8.3), and the owned numeric plane (§6.8, §8.4) exist.

The trusted checker validates fully elaborated declarations against the type theory: definitional equality with beta, eta (for functions *and* structures), delta (unfolding, ordered by definitional height), zeta (`let`), iota (recursor computation rules), definitional proof irrelevance in `Prop`, quotient computation (`Quot.lift` on `Quot.mk`), and native reduction rules for `Nat` and `String` literals (the Reference accelerates kernel `Nat` arithmetic — add/mul/div/mod/pow/gcd/comparisons and friends — through GMP so `decide`-style proofs are feasible). Inductive families are checked for strict positivity, universe validity, and valid recursor/eliminator generation, with support for mutual and nested inductives (nested compiled to mutual), reflexive types, and K-like reduction for the subsingleton eliminators. Well-founded and structural recursion are *elaborator* constructions — the kernel sees only recursors, `WellFounded.fix`, and their unfold equations; `partial`/`unsafe` definitions bypass the logic and are marked. The axioms trusted by the ecosystem are exactly three: `propext`, `Quot.sound`, `Classical.choice`. The kernel also enforces environment-level rules: no duplicate names, universe-parameter discipline, projection/structure-eta coherence, and the reducibility-irrelevant guarantee that kernel acceptance never depends on elaborator hints. ↷ *We keep every judgment bit-for-bit and replace the engine: a dual-engine Crucible with certificates, fuel parity, and a size covenant* (§8).

### 1.3 The elaboration plane

Elaboration turns surface syntax into kernel terms through a monadic tower (`CoreM → MetaM → TermElabM/TacticM`) over a metavariable context with delayed assignments, postponement, and backtrackable state. The load-bearing machinery: higher-order unification with first-order and Miller-pattern approximations; transparency-aware `whnf`/`isDefEq` with unfolding controlled by reducibility attributes and definitional heights; type-class resolution (`synthInstance`) with tabling, instance priorities, `outParam`/`semiOutParam`, default instances, and a result cache; coercion insertion; structure-instance and anonymous-constructor elaboration; dependent pattern-match compilation into auxiliary matchers with generated equation lemmas and splitters; `deriving` handlers; the attribute system; `variable`/section/namespace/`open` scoping with ambiguity resolution; auto-bound implicits; and the `Elab.async` machinery that (as of the pin) parallelizes parts of elaboration with documented result instability. Options thread through everything, with `maxHeartbeats` (default 200000, counted in allocation-linked units of thousands) and `maxRecDepth` (default 512) as the canonical resource limits, plus the newer resource-limit surface and its `unlock_limits` escape. ↷ *We keep the semantics, the option surface, and the limit accounting; we replace the scheduler (deterministic dataflow, §10.6), the instance engine's internals (Synod, §10.4), and the diagnostics substrate (Palimpsest, §15).*

### 1.4 The syntax plane

The parser is a Pratt-style extensible engine over token tables, producing `Syntax` trees whose node kinds, source-info discipline (original vs. synthetic positions, leading/trailing whitespace attachment), and antiquotation structure are all API surface for the macro system. Hygiene is implemented by macro scopes appended to names; `syntax`, `macro_rules`, `elab`, notations, and custom syntax categories let libraries extend the grammar at every precedence level, which means *the grammar is not static*: parsing a file is interleaved with elaboration, because a command can install parsers used three lines later. Error recovery exists but is uneven; incremental reparse operates at command granularity keyed by the server's snapshot machinery. ↷ *Quill preserves node kinds, positions, hygiene observables, and the parse/elab interleaving contract exactly; recovery quality and incremental granularity are ours to improve* (§9).

### 1.5 The metaprogram & compilation plane

Lean code — including every tactic in mathlib — is compiled through the Lean compiler pipeline (to LCNF, with specialization, simplification, closure conversion, reference-count insertion with borrow inference and constructor-reuse analysis, boxing) and then either interpreted or emitted as C and compiled by a system toolchain into the executable/`--precompileModules` shared libraries. The interpreter is the default execution vehicle for user tactics not precompiled, and its overhead is a well-known tax on proof automation. `@[extern]` binds Lean declarations to C symbols; `@[implemented_by]` swaps logical definitions for efficient ones at the boundary the ecosystem has learned to respect; `initialize`/`builtin_initialize` run effectful module initialization at import/load; `native_decide` evaluates closed propositions through the compiled path, explicitly enlarging the trusted base to the whole compiler and runtime. ↷ *Golem keeps the observable semantics — evaluation results, effect ordering, initialization timing, the `@[extern]`/`@[implemented_by]` contracts — and replaces the vehicle: an ABI-valued register VM with inline caches and an owned baseline JIT, plus the Iron native backend so `lake build` needs no C compiler* (§11, §17.3).

### 1.6 The object & module plane

The runtime object model (`lean.h`) is the ABI the entire native ecosystem compiles against: a fixed object header packing reference count, a memory-kind/other byte, and a tag byte; constructor objects with scalar-field packing; arrays, scalar arrays, strings; closures; thunks, tasks, `IO.Ref`s; external objects with class vtables; big-number objects backed by GMP; small `Nat`s boxed as odd tagged pointers (`(n << 1) | 1`); and a tri-state reference-count regime (single-threaded, atomic/shared, and persistent objects that are never counted). Compacted regions serialize whole object graphs as relocatable memory images — and **`.olean` files are exactly that**: a small header (magic, versioning, base address) plus a compacted region containing the module's `ModuleData` object graph (imports, constants, environment-extension entries, and friends), loaded by mmap-plus-relocate. This is why import is fast, why olean files are platform-coupled pointer dumps, and why arbitrary environment-extension state (simp sets, instance caches, mathlib's own extensions) round-trips without running code: it is all just objects in the region. ↷ *Marrow implements the object model exactly, with layout constants extracted from the pinned `lean.h` into generated conformance tables rather than transcribed by hand; Grimoire implements the region codec for byte-compatible read* and *write, then adds an additional content-addressed module format alongside* (§6, §7).

### 1.6b The float plane (a weakness worth its own line)

`Float` arithmetic is IEEE-754 binary64 in hardware (portable), but the transcendental family (`Float.sin`, `exp`, `log`, `pow`, …) defers to platform libm — so compiled evaluation, `#eval`, and `native_decide`-class computation drift across glibc/macOS/Windows in the last ULPs, and even the Reference cannot promise itself cross-platform bit-stability. ↷ **fln-libm** (§6.8): owned, fixed-implementation transcendentals, bit-locked across the certified matrix, with the divergence-from-any-one-platform's-libm recorded as a Behavior Note rather than pretended away.

### 1.7 The tooling plane

`lean --server` is a watchdog that spawns a worker process per open file; each worker imports the file's dependencies (for a mathlib file: ≈60 seconds and multiple GB of RSS, per worker), elaborates by command-granular snapshots with sub-declaration incrementality inside supported proof structures, and serves LSP plus the Lean-specific extensions: `$/lean/plainGoal`, `$/lean/plainTermGoal`, the `$/lean/rpc/*` session protocol that powers the InfoView, `textDocument/waitForDiagnostics`, semantic tokens, and the widget system (user widgets as client-executed JavaScript modules delivered over RPC). The InfoTree — elaboration information recorded per syntax range — is the substrate of hover, goals, and go-to-definition. ↷ *Lantern speaks the identical wire dialect from a single daemon with one shared immutable import heap, killing the per-worker multiplication; the InfoTree observables are preserved; the internal topology is not* (§14).

### 1.8 The build plane

Lake is the package manager and build system, itself a Lean program: `lakefile.lean` is executed to produce the package configuration (with `lakefile.toml` as the declarative subset), targets and facets form the build graph, `require` fetches git dependencies, Reservoir is the registry, and `lean-toolchain` files drive `elan`'s toolchain multiplexing. Mathlib's cloud cache is a separate executable (`lake exe cache`) downloading prebuilt oleans keyed by file hashes. Invalidation is file-granular over the import DAG: editing a leaf invalidates its entire transitive dependent cone at file resolution, which is the single largest quality-of-life complaint in the ecosystem and the direct cause of the cache's existence. The module system (in active development at the pin: `module` keyword, public/private import boundaries) is the Reference's own acknowledgment that file-granular semantic exposure is the bottleneck. ↷ *The Ledger implements the Lake surface bit-compatibly — `lakefile.lean` executes on Golem — over a declaration-granular content-addressed store, making both the cache and the module system's motivating problem largely moot, while still honoring `module` boundaries as authored* (§13).

### 1.9 The library reality

`Init` and `Std` ship with the toolchain and are imported by everything; Batteries and mathlib define the ecosystem's real behavioral envelope. Mathlib is the Corpus: over 115,000 definitions, over 232,000 theorems, 1.5M+ lines, thousands of `simp` lemmas, its own environment extensions, its own tactics (many now migrating onto core's `grind` infrastructure), and a monthly toolchain-migration treadmill that pins every downstream project to specific Reference versions. Any implementation that does not elaborate the Corpus is not an implementation of Lean; the Corpus is therefore both our hardest test and our most honest specification, and G4 (§22.3) treats it as such.

### 1.10 The trust chain, stated plainly

What one must trust today to believe a mathlib theorem: the C++ kernel (unverified), the C++ runtime and GMP (arbitrary-precision arithmetic *inside kernel reduction*), the compiler and system C toolchain if `native_decide` or precompiled plugins are involved, CaDiCaL and its LRAT path if `bv_decide` is involved, and the olean loader's pointer-relocation logic. External checkers exist — lean4checker (re-checks environments with the Reference kernel), lean4lean (a Lean-verified re-implementation of the kernel rules), nanoda and friends (independent kernels) — but none is in the default loop, none covers the object codec, and none is a full system. ↷ *FrankenLean's posture: shrink what must be trusted (Crucible's covenant, owned bignum, owned SAT proof checking), make the rest checkable (certificates, export, differential rigs), and never let performance machinery into the trusted base (every Anvil engine emits kernel-checked artifacts).*

### 1.11 The weakness census

The specific, load-bearing pains this program exists to delete — each carried forward into a bet, a subsystem, or a Behavior Note:

1. **Invalidation at file granularity.** A leaf edit re-elaborates the file cone, not the dependency cone. Hours where seconds are warranted; a cloud cache invented to paper over it. → B1, §13.
2. **The per-worker import multiplication.** ≈60 s and GBs of RSS to attach mathlib, paid per open file and per search branch; the documented dominant cost of 2026's automated-proving pipelines. → §14.2.
3. **Re-elaboration as the price of a proof state.** 18–735 s to reconstruct a theorem context for search; no cheap snapshots across processes. → §14.4, §16 (Envoy).
4. **Schedule-dependent elaboration.** `Elab.async` can change results; determinism is aspirational, not contractual. → B4, FL-INV-01.
5. **The interpreter tax.** Un-precompiled tactics run interpreted at a large constant factor; `precompileModules` trades it for C-toolchain friction and plugin ABI fragility. → §11.
6. **The bootstrap treadmill.** stage0 updates in nearly every nightly; deep changes pay a multi-stage dance; the system cannot be built from sources without a prior self. → commitment 3.
7. **The trusted-base sprawl.** C++ kernel + GMP + (conditionally) C toolchain + CaDiCaL. → B3, B5, §8, §12.5.
8. **Object-format opacity and platform coupling.** Pointer-dump oleans: fast, but unmergeable, un-diffable, un-addressable, and bound to the producing toolchain's layout epoch. → §7.3–7.4.
9. **Diagnostic opacity.** Unification and instance failures reported as end states, not explanations; no first-class "why" for the two most common failure classes. → §15.
10. **The toolchain-migration treadmill.** Monthly releases; whole-world invalidation per bump; every project pinned. → §13.6 (epoch-stable caching), §22.5 (epoch policy).
11. **Search and discovery as an afterthought.** Premise selection and library search live outside the toolchain (external services, ad-hoc indexes), while the toolchain holds all the data. → B6, §16 (Bloodhound).
12. **The external-tool contract.** git, cc, CaDiCaL, and (for docs) a browser-side math stack — each a real install-base and CI tax. → the external-tools policy above; §16 (Folio).

## 2. Foundation audit: the FrankenSuite substrate

Audited at the Rust-native symbol level (Rule D6), with the whole suite commit-pinned in `SUITE.lock` (§2.10). The mapping below is the reuse plan that makes the leapfrog credible: *specific existing assets* to *specific FrankenLean subsystems*.

**2.1 asupersync — the operating system.** Structured concurrency with region-owned tasks, obligations, and an `Outcome` lattice; `Cx` capability contexts with narrowing; a three-lane scheduler; cancel-correct channels, combinators, and supervision; the lab runtime (virtual time, seeded deterministic scheduling, DPOR-style schedule exploration, chaos injection, trace capture/replay, crashpacks); a formal layer (Lean-checked invariants — yes, the suite's async runtime already carries Lean proofs, a symmetry this program intends to enjoy — and TLA+ export discipline); RaptorQ fountain coding with per-symbol authentication; the full networking stack (TCP/UDP/QUIC/HTTP/1.1/2/3/WebSocket/TLS/DNS/gRPC); macaroon capability tokens; region heaps with generational handles and epoch reclamation; io_uring-backed fs/VFS. **Used by:** every subsystem. Elaboration of one declaration *is* a region; cancelling a stale server request drains its region deterministically and frees its heap; the native machinery and Golem-hosted user metaprograms share one cancellation tree; Lantern's transport is asupersync's HTTP/WS/JSON-RPC plumbing with zero protocol code written here; the Tribunal's differential and torture rigs run the entire toolchain under the lab runtime, seed-replayable. The deep synergy is the same one FrankenGraphDB named: obligations map onto prover invariants (an unassigned metavariable is an obligation; a pending kernel check is a two-phase publication; an unsatisfied instance goal is a lease), so *FrankenLean is a prover written in the asupersync programming model*.

**2.2 frankensqlite — the durable store.** A ground-up Rust SQLite reimplementation with concurrent writers, MVCC, and information-theoretic durability (RaptorQ-protected). Unlike FrankenGraphDB — which forked designs because graph objects are not pages — FrankenLean's durable workloads *are* relational-plus-blob workloads, so we **link frankensqlite directly as an embedded database**: the Ledger's CAS metadata and dependency edges, the build-event journal, the Tribunal's evidence store, Bloodhound's index shards, Palimpsest's trace archives, and the telemetry rollups all live in single-file databases with WAL durability and honest crash semantics. A `fln doctor --sql` REPL over the build database (§20) falls out for free. Blob payloads (compacted regions, certificates) live in the CAS value store beside it, RaptorQ-armored by asupersync's coder for the archival tier.

**2.3 franken_networkx — the graph brain and the determinism doctrine.** Two distinct gifts. (a) Algorithms over the graphs a prover actually has: the declaration dependency DAG (topological scheduling, critical-path analysis for build planning, dominators for invalidation-cone computation, SCCs for mutual-block detection), the import graph, instance graphs (cycle diagnostics in Synod), and the call graphs Golem's specializer consults. (b) **CGSE** — the Canonical Graph Semantics Engine posture: named tie-break policies, complexity witnesses, and a witness ledger, generalized here from graph algorithms to *elaboration itself*: wherever an order is semantically free (parallel merge order, cache iteration, error aggregation), a registered policy pins it, and the pinned choice is part of the reproducibility closure (FL-INV-01, §18.7). This is Bet B4's doctrinal ancestor.

**2.4 frankensearch — the discovery engine.** Two-tier hybrid search: sub-millisecond lexical/embedding first pass (potion-class small embedder, CPU-only) with quality-refined reranking. **Used by:** Bloodhound (§16.1) — library search, `exact?`/`apply?`-class premise retrieval, and the MCP `search_lemmas` tool — indexing the *live environment* (statements, docstrings, simp-set membership, namespace structure) incrementally as the Ledger commits declarations. The embedder is bundled; no network, no Python, no service dependency. Determinism note: ranking is standard-mode UX; anything that feeds a proof term passes through the kernel, so search can be aggressive without touching the trust story.

**2.5 frankentui — the terminal surface.** Diff-based rendering, inline mode, RAII cleanup. **Used by:** the `fln` CLI's build progress (live dependency-cone visualization during Ledger builds), the terminal InfoView (`fln goals` attaching to Lantern over the same RPC the editor uses — goals, hover, and widgets-as-text in a tmux pane, which is precisely where this ecosystem's agents already live), and Tribunal dashboards.

**2.6 franken_markdown (+ fmd-font, fmd-math) — the document plane.** The typography stack with bundled Computer Modern and the native TeX-math layout engine built for FrankenManim. **Used by:** Folio (§16.2), the doc-gen4-compatible documentation generator: docstring markdown, `$…$` mathematics rendered natively to HTML and PDF, syntax-highlighted Lean via a Quill-fed lexer, cross-reference graphs via franken_networkx, zero browser-side math stack. Mathlib's documentation build — today a heavyweight separate pipeline — becomes a Ledger facet.

**2.7 fastmcp_rust — the agent protocol.** Cancel-correct MCP server framework with zero-copy serialization. **Used by:** Envoy (§16.3), the first-class agent surface: tools for goal inspection, tactic application against snapshots, premise search, `#eval` under budgets, certificate retrieval, and Ledger queries — the operations 2026's proof-search papers currently obtain by scraping an LSP meant for humans.

**2.8 atp — the transfer plane.** Fountain-coded file transfer (RaptorQ over UDP/QUIC) that outruns tuned rsync on real networks, verified end-to-end. **Used by:** Ledger cache federation (§13.5): CAS sync between developer machines, CI, and cache servers, content-addressed and signature-verified, replacing the mathlib cache's HTTPS-tarball model with symbol-oriented multi-donor pulls. Falls back to asupersync's HTTP client where atp is not deployed.

**2.9 Optional tier.** `frankentorch`: learned tactic/premise ranking experiments behind a feature flag, never on a gate's critical path, never affecting certified artifacts. `franken_node`: a server-side widget-JS host so terminal and Folio surfaces can render interactive widgets — exploratory, wire-compatible pass-through remains the contract (§16.4). Neither appears in the default build.

**2.10 SUITE.lock.** Exact commits/checksums for every foundation repo, the rustc nightly, and target-feature sets; CI builds only from the lock; upgrades are deliberate and Tribunal-diffed. The Reference pin (`v4.32.0`) and the Corpus pin live in the same lock with the same ceremony.

**2.11 The upstream contributions ledger.** Primitives that belong in a foundation crate land there, tracked in `UPSTREAM_LEDGER.md`. Seed entries: asupersync — a deterministic work-stealing *merge* combinator (ordered join of speculative results) if the existing quorum/pipeline combinators don't already express it; franken_networkx — incremental dominator maintenance for dynamic DAGs (the invalidation cone under live edits); frankensearch — incremental index commit hooks matching the Ledger's transaction boundaries; franken_markdown — the Lean lexer profile for the shared highlighter; frankensqlite — any blob-store access pattern the CAS proves out. fln-bignum stays in-repo (kernel-adjacent code answers to the kernel's covenant), with an explicit revisit trigger if a second suite member needs arbitrary precision.

## 3. The dependency & safety doctrine

**Rule D1 — the governed closure.** Authoritative `fln-*` crates introduce **no new unreviewed direct runtime dependencies**. The complete transitive closure — the suite's own dependencies, proc-macros, build scripts, platform crates — is pinned and governed by an explicit per-package allowlist (name/version/source/checksum, features, license, build-script and proc-macro status, native-link status, unsafe-audit status, reason, owner, upgrade policy), with runtime / build / dev / fuzz tracked under separate policies. The audit is an owned `fln-conformance` check; CI fails on any unlisted package. No serde, no tokio, no rocksdb, no LLVM, no cranelift, no gmp-sys, no external SAT solver: what §21.2 does not list as built-in-house is not in the program.

**Rule D2 — two inherited external tools, both optional.** A system C compiler may be invoked *only* by the `--backend c` code generator, mirroring the Reference's own `leanc` contract for users who want C-level interop or platforms Iron does not yet target; the sovereign path (`--backend iron`) requires none. System `git` may be invoked *only* by Lake-compatible dependency fetching, mirroring upstream Lake, under the full subprocess security protocol (argv-only invocation, per-job private temp dirs, timeouts with process-tree cancellation, output-size limits, environment allowlist, atomic result publication, executable path *and content hash* into provenance). Their absence yields **capability errors naming the alternative** — never silent substitution. Nothing else is ever spawned: no cc at check time, no CaDiCaL, no curl, no tar, no LaTeX. `--reproducible` artifact sets exclude both tools' products by construction. The Reference toolchain is *not* a third tool: it is Tribunal apparatus only, and a mechanical release check proves no `fln` binary can locate, spawn, or link it (D8).

**Rule D3 — the unsafe posture.** `#![forbid(unsafe_code)]` in every authoritative crate. Project-authored `unsafe` exists only in three named boundary crates, each with `deny(unsafe_code)` at the root and narrowly scoped, ledgered `allow` sites: `fln-unsafe-abi` (Marrow's object layout, tagged-pointer arithmetic, compacted-region relocation, `dlopen` of Reference-ABI plugins, the exported `lean_*` symbol surface), `fln-unsafe-region` (mmap and arena primitives not already provided by asupersync's region heaps), and `fln-unsafe-jit` (Iron-JIT's executable-page management, feature-gated, standard-mode only). Every unsafe operation gets a ledger row: path, invariant, evidence, safe fallback, and no-claim boundary; CI rejects an unledgered site. The kernel crate (`fln-kernel`) is `forbid(unsafe_code)` — the trusted computing base contains zero project-authored unsafe, full stop. Two structural laws complete the geometry: **(a) no inward calls** — no `fln-unsafe-*` crate may depend on `fln-kernel` or `fln-checker`, so ABI unsafety can never call into the trust base with unchecked values; **(b) no unsafe path to admission** — no unsafe crate exports any function whose return type is, or can be laundered into, a checked declaration. CI walks the dependency graph and the export surface for both.

**Rule D4 — explicitly not built here.** Async runtime, scheduling, networking, deterministic lab, fountain codes, capability tokens — asupersync. Durable relational storage — frankensqlite. Graph algorithms and the determinism doctrine's machinery — franken_networkx. Search and embeddings — frankensearch. Terminal rendering — frankentui. Markdown, fonts, math typesetting, highlighting, PDF — franken_markdown. MCP — fastmcp_rust. Bulk transfer — atp. FrankenLean's owned surface: the parser/macro engine, the elaborator, the kernel, bignum, the metaprogram compiler/VM/JIT, the runtime-ABI twin, the module codecs, the build fabric's logic, the server's session logic, the rewrite/decision engines and their proof producers, the diagnostics substrate, and the conformance apparatus.

**Rule D5 — the clean-reference posture (licensing and provenance).** The Reference is Apache-2.0; reading it is legal, encouraged, and *required* — this is a semantics-fidelity program, not a black-box guessing game. The discipline is provenance, not blindness: no file is translated wholesale; behavior is specified from the pin into Tribunal fixtures and then implemented against the fixtures; the generated conformance tables of §6.5/§7.2 (`ABI_CONTRACT.md`, `OLEAN_CONTRACT.md`, the extern census) are *extracted mechanically from the pin* with their extraction scripts checked in, so layout constants are never hand-copied and never silently drift; NOTICE files carry attribution for the Reference and for every adopted fixture. Vendored upstream Lean *sources* (`Init`, `Std`, `Lean`) are held byte-identical at the pin under their license, never forked, and used only in D8's three capacities: Tribunal corpus, census-extraction input, and — for the library-code partition — *source input* our native toolchain elaborates like any user library (§4.3). They are never executed as toolchain implementation.

**Rule D6 — symbol-level binding proof.** Every dependency-map and displacement-map row names the exact Rust symbol bound, links a fixture, and states its determinism class. Headline conformance percentages are never accepted as evidence; the Parity Ledger (§18.1) is row-per-symbol or it is marketing.

**Rule D7 — claims have types.** (See below.) Alongside the claim types, every operation in the program carries a **determinism class**: `D0 mathematical` (unique result, bit-identical), `D1 canonicalized` (multiple valid paths, one CGSE-pinned output), `D2 replayable` (seed/schedule-dependent, receipt reproduces it), `D3 advisory` (search/ranking may vary; identity recorded; correctness never depends on it), `D4 external` (subprocess/plugin behavior not controlled; capability-gated and boundary-recorded). Kernel checking is D0/D1; faithful-mode diagnostics are D1; Bloodhound ranking is D3; a user `@[extern]` call is D4. Cache keys, receipts, and the Parity Ledger all carry the class.

**Rule D7 (continued) — the claim types.** Every load-bearing statement in this program is registered as exactly one of: `invariant` (FL-INV-*: must hold on every execution), `proof` (theorem within a named formal model, in the asupersync house style), `bounded_model` (checked within declared bounds), `statistical` (confidence under named assumptions), `slo` (empirical target), or `benchmark` (measured on a pinned fixture). A weaker class may inform policy but may never enforce or justify a stronger one. Seed invariants: **FL-INV-01** schedule independence (same input closure ⇒ identical environment, diagnostics set, and artifacts at any thread count); **FL-INV-02** kernel soundness locality (no code outside `fln-kernel` can cause acceptance of a declaration the kernel rejects); **FL-INV-03** Mirror fidelity (every builtin-census row's native implementation is observationally equivalent, per its registered equivalence class, to the pinned upstream definition it replaces — established and continuously re-established by the Tribunal's differential rigs against the oracle, §18); **FL-INV-04** codec fidelity (Grimoire read∘write is identity on the byte level for `faithful`-mode oleans); **FL-INV-05** cache soundness (a Ledger hit is semantically identical to a recomputation); **FL-INV-06** untrusted engines (no Anvil engine's output enters an environment without a kernel-checked artifact); **FL-INV-07** inconclusive-is-not-rejected (resource exhaustion, cancellation, and internal faults yield typed `Inconclusive`/`InternalFault` outcomes that are never rendered as, cached as, or promoted to acceptance *or* rejection).

**Rule D8 — the Oracle-Only Law.** The Reference implementation — its binaries, its C++ sources, its self-hosted `Lean.*` sources — participates in this program in exactly three capacities: (1) **differential oracle** inside the Tribunal (its binaries run in isolated harness processes and its verdicts/artifacts/transcripts are compared against ours); (2) **fixture and census mine** (checked-in extraction scripts derive the ABI/olean/wire/builtin contracts and golden corpora from the pin — including an instrumented, build-time-only oracle variant that dumps unifier decisions, instance-search traces, and macro expansions as golden trace fixtures, §18.3); (3) **source input** — the vendored `Init`/`Std` `.lean` files are *data* that our native toolchain elaborates, exactly as it elaborates mathlib; upstream-authored source compiled by our compiler is input, not implementation reuse. Nothing else. A development build may additionally run a lockstep diagnosis harness (§18.10) whose every product is `ORACLE_FALLBACK`-tagged, cache-inadmissible, and gate-inert; release profiles compile the harness out and CI proves its absence. **Typed error taxonomy (normative):** errors cross crate boundaries as typed, versioned values — `SyntaxFailure`, `MacroFailure`, `ElaborationFailure`, `KernelRejection`, `KernelInconclusive` (resources/cancellation — FL-INV-07), `ArtifactCorrupt`, `ArtifactEpochMismatch`, `AbiViolation`, `CapabilityDenied`, `PluginCrashed`, `BuildFailure`, `ProtocolFailure`, `ReplayDivergence`, `InternalInvariantViolation`. Faithful-mode frontends render these into Reference-compatible text/JSON; receipts retain the typed cause. Panics are invariant failures, never user diagnostics; malformed source, artifacts, protocol messages, and plugin output must not panic.

**4.2 The three modes.** One type theory, one kernel, always; the modes govern everything *around* it.

- **`faithful`** — bug-for-bug observational parity with the pin for everything the surfaces above can see, including **fuel parity**: the heartbeat-counting law is replicated at its allocation-linked granularity so `maxHeartbeats` timeouts and `maxRecDepth` failures fire on the same inputs (a deliberately deep flex of the determinism doctrine — the G0-6 spike calibrates it, and Behavior Notes record any residual divergence class). This is the mode CI farms and migration audits run; it is also the mode the Tribunal's differential rigs pin.
- **`sound`** (default) — the same language and the same accept/reject verdicts, with documented improvements where the Reference's behavior is an artifact rather than a meaning: deterministic parallelism always on, Palimpsest diagnostics, Ledger incrementality, better error recovery, stable-across-thread-count everything. Every divergence from `faithful` is a Behavior Note with migration guidance; *none* may change which well-formed declarations exist or what they mean.
- **`frontier`** — the extension surface: olean-next emission, e-graph simp lanes, speculative tactic portfolios, MCP write-tools, Iron-JIT. Frontier features never leak into `faithful`/`sound` artifacts; a frontier artifact is marked as such in provenance.

**Determinism axis (orthogonal).** `standard`: deterministic given the input closure on a given build (FL-INV-01 always holds). `certified` (`--reproducible`): the complete content-hashed input closure — sources, toolchain, SUITE.lock, options, plugin hashes, mode — yields bit-identical artifacts across the certified platform matrix, with a sidecar manifest; products of the two D2 tools are excluded from certification by construction. The D0-D4 operation classes of D7 refine both settings.

**Evidence axis (orthogonal, per surface and per release).** Compatibility is never a percentage. Every Parity-Ledger row carries a level: **L0 recognized** (inventoried; precise `unsupported` result, never a guess) -> **L1 shape-compatible** (names/shapes exist; development only) -> **L2 behavioral** (gated corpus passes; exclusions explicit) -> **L3 differentially closed** (source-derived + generated + fuzz + metamorphic + real-project differentials, no unclassified divergence) -> **L4 drop-in attested** (L3 plus unchanged downstream projects, native modules, and editors pass on the platform). A release declares per-epoch/per-platform levels **R0 research** -> **R1 checker preview** -> **R2 project-local toolchain** -> **R3 ecosystem candidate** -> **R4 drop-in epoch replacement** (all mandatory rows L4, no fallback code) -> **R5 hardened replacement**. FrankenLean 1.0 means R4 for the pinned epoch on the declared platform matrix — not "many tests pass".

**4.3 The Native Mirror (Bet B2, normative).** The state is one — Marrow objects, Grimoire environments — and the implementation is one: ours. The builtin namespace at the pin (every symbol in `Init`/`Std`/`Lean` that user metaprograms, compiled code, or the wire can reach) is mechanically partitioned by the census of Appendix C into exactly three classes, and every symbol lands in one:

- **Toolchain-API (native).** The machinery of the language — `Lean.Meta.whnf`/`isDefEq`/`synthInstance`, the `CoreM`/`MetaM`/`TermElabM`/`TacticM` services, elaborator entry points, environment and extension access, `Syntax`/`Expr` construction and inspection, the compiler API, IO/Task/ST, `Nat`/`String`/`Array`/`Float` primitives, the `@[extern]` set — implemented **in native Rust** inside Quill/Athanor/Crucible/Golem/Anvil/Marrow and registered under the upstream names behind a **generated façade**: adapter stubs derived from the census (upstream signature, universe/binder telescope, effect class) whose bodies call the native services. When a mathlib tactic calls `Lean.Meta.whnf`, it executes our unifier. Every façade row states: symbol; upstream semantics anchor (source location at the pin); equivalence class (`bit` — identical results always; `obs` — identical observable behavior, internal traces may differ; `perf` — identical results, different cost); evidence (differential fixtures against the oracle, golden traces from the instrumented oracle of §18.3, property tests, fuzz budget); determinism class (D7); and its Parity-Ledger L-level.
- **Library-code (source-elaborated).** Pure upstream-authored library content with no implementation role — `List.map`, `Std` data structures, lemmas — is not reimplemented and not hosted: the vendored `.lean` **sources** at the pin are *elaborated by our native toolchain* exactly as mathlib is, compiled by Golem where executed, checked by Crucible like everything else. Upstream-authored source as input is the same relationship FrankenSQLite has to SQL files; it is data, not implementation reuse (D8-3).
- **User-code (VM-executed).** Everything downstream — mathlib's tactics, `aesop`, project macros, lakefiles, `#eval` bodies, plugins — runs on Golem against the toolchain-API surface. This class exists because executing user metaprograms *is the Lean language*; there is nothing "legacy" about it.

The partition is the load-bearing inversion of the hosted-frontend designs this program rejects: compatibility of the Meta surface comes from a census-complete native façade proven equivalent differentially, not from running upstream's implementation and calling the hosting a plan. Three laws govern it. **(1) Census totality:** no unclassified symbol; a symbol the census cannot classify blocks the surface's L-level, never gets guessed. **(2) Fidelity by evidence (FL-INV-03):** every toolchain-API row is proven against the oracle — fixture differentials, golden decision traces, and the metaprogram-corpus rig of §18.2 that elaborates the ecosystem's real tactic code through the façade and diffs environments, InfoTrees, and diagnostics against the Reference. Where a Meta API intrinsically exposes internal object identity or allocation order, the row is classified at its honest L-level with a Behavior Note rather than hand-waved. **(3) No hidden oracle (D8):** the fallback for a façade defect is fixing the native implementation against the fixtures — there is no "run the upstream definition instead" switch in any release binary; the development-only lockstep harness (§18.10) exists for *diagnosis*, its products are ORACLE_FALLBACK-poisoned, and it is compiled out of releases. The contract's punchline survives the inversion: mathlib's tactics do not know or care that every builtin beneath them is new — and the Tribunal can prove it, symbol by symbol, on demand.

## 6. Marrow: the runtime & ABI twin (fln-rt, fln-unsafe-abi)

Marrow is the load-bearing act of drop-in compatibility: the Reference's object model, implemented exactly, in Rust, exported under the `lean_*` symbol names.

**6.1 The object model.** The header layout (reference-count word, memory-kind/other byte, tag byte), constructor objects with the Reference's scalar-field packing rules, arrays and scalar arrays, strings (UTF-8 with byte-length/capacity semantics and the Reference's position conventions), closures (arity, fixed-args), thunks, tasks, refs, external objects with class vtables, and bignum objects — every category, with layout constants taken from the generated contract of §6.5, never from memory or transcription. Small `Nat`s box as odd tagged pointers exactly as upstream (`(n << 1) | 1`); boxing/unboxing of scalars follows the Reference's rules per type so compiled foreign code agrees about every bit.

**6.1b Two heaps, one membrane.** The ABI object model above is the **CompatHeap**: exact layouts, pointer identity, RC discipline — everything foreign code and user-metaprogram values can observe, allocated and freed only through membrane APIs, with every unsafe operation confined to `fln-unsafe-abi`. Behind it lives the **NativeHeap**: the toolchain's *own* working structures — Quill trees, Athanor transactions, Crucible terms, Ledger records — as typed handles in asupersync region heaps with generational IDs, persistent/immutable where semantics allow, and no obligation to the ABI's layout. Values cross domains only through generated converters that state ownership and failure semantics; conversion is lazy and boundary-specific (a metaprogram that never inspects a term never forces its Compat projection). Two laws: a raw CompatHeap pointer never enters Crucible (the kernel checks NativeHeap terms decoded through the typestate gate of §8.2b), and no source-level structure is duplicated across both heaps except at a live ABI crossing — the memory-doubling risk is a named gate (PG-6), not an accident. This split is a dividend of the Oracle-Only Law: because no upstream implementation code runs inside the toolchain, the elaborator's state no longer has to be ABI-shaped, and only the genuinely user-visible surface pays the compatibility tax.

**6.2 Reference counting, tri-state.** Single-threaded fast-path counts, atomic counts for shared objects, and persistent objects (region residents and marked immortals) that are never counted — matching the Reference's discipline so that ABI-crossing code that borrows/increments/decrements behaves identically. Golem's compiled code uses the same borrow-inference and constructor-reuse disciplines the Reference's compiler established (§11.3), because FBIP-style reuse is observable through performance cliffs the ecosystem has tuned against. Cycle-freedom is structural (pure values); `IO.Ref` and task graphs follow the Reference's ownership story. Hardened builds add debug ownership shadows: double-release and foreign-pointer misuse detection with provenance tags, quarantine instead of corruption, and deterministic replay events for ownership faults — divergence-receipted, never silently repaired.

**6.3 The allocator.** An owned size-classed allocator in the mimalloc tradition (thread-local pages, size-class bins, deferred frees), because the Reference's allocation behavior is *semantically visible* through heartbeat accounting (§4.2's fuel parity) and performance-visible everywhere. The heartbeat hook lives here: one counting law, calibrated in G0-6, shared by both engines.

**6.4 Compacted regions.** Region construction (object-graph serialization with pointer relocation tables), region loading (mmap, relocate-or-copy fallback, page-sharing across consumers — the enabler of Lantern's shared import heap), and region hygiene (immutability enforcement; persistent-RC marking). This is deliberately *the same code path* Grimoire's olean codec uses, so olean fidelity and runtime-region fidelity are one test surface.

**6.5 The generated ABI contract.** A checked-in extraction script parses the pinned `lean.h` (and the runtime sources where layout is defined) and emits `ABI_CONTRACT.md` + a Rust constants module: header field offsets and widths, tag values per object category, size-class and scalar-packing rules, the full exported `lean_*` function census with signatures. CI re-runs the extraction against the pin and fails on drift between the contract, the constants module, and the handwritten shims. Rule D5 in action: the ABI is *derived*, not remembered. Every exported symbol additionally carries a status — `NativeSafe` (safe Rust below a generated wrapper), `RawPlatform` (audited OS/ABI operation in a membrane capsule), `CompatWrapper` (exact ABI wrapper over a native operation), `ReferenceSemanticAdapter` (preserves an epoch quirk), or `Unsupported` (blocks L4 for the platform) — there is no unclassified symbol and no call-through to the Reference anywhere. **Panic law:** no Rust panic crosses an exported ABI boundary; every wrapper catches, converts only what the epoch contract maps, and otherwise isolates or terminates per policy — a wrapper never fabricates a successful Lean result after an internal panic.

**6.6 The plugin doors.** Loading: Reference-built shared libraries (`precompileModules` artifacts, user `@[extern]` libraries) `dlopen`ed into FrankenLean, their `lean_*` imports resolved against Marrow's exports, their initializers run under Golem's initialization ordering (§11.5). Serving: FrankenLean-built plugins loadable by the Reference — the reverse direction is the harder promise and gets its own Tribunal matrix (§18.5). Sandboxing posture: a plugin is native code and is trusted as such; what FrankenLean adds is *provenance* (hash and origin in the build record) and the capability to run `--no-plugins` elaboration for audit builds, with any plugin-dependent declaration flagged. Loading is **revocation-first** in `sound` mode: hash and trust-card check before `dlopen`, never after; and each load emits an initialization-DAG receipt so cycles, duplicate initializers, and late capability requests are explainable rather than opaque loader failures. The strongest standing ABI rig is the **stage0 gauntlet** (Tribunal, §18.5): the Reference's own stage0-generated C, compiled by the optional D2 cc *as test apparatus*, linked against Marrow's exports, running the Reference's runtime test suite — if the membrane is wrong anywhere, upstream's own code says so, loudly, early.

**6.7 Effects.** IO, timers, file system, process, and task pools ride asupersync behind the Reference's `IO`/`Task` semantics: `Task.spawn`/`bind`/`get` observable ordering, `IO.Ref` atomicity, `IO.Process` behavior, and the interruption/`checkSystem` polling contract preserved; the vendored-libuv event loop is displaced wholesale. Under the lab runtime the entire effect plane virtualizes, which is how the Tribunal replays server sessions and torture-tests cancellation. In `faithful` mode the default ambient authority approximates an ordinary Lean process (drop-in means unchanged packages just run, with effects *observed*); `sound` mode can require declared capabilities and fail closed (§11.5).

**6.8 fln-libm: owned deterministic transcendentals.** `Float` arithmetic is hardware IEEE-754 and portable; the transcendental family is where upstream leaks platform libm into observable results. FrankenLean ships an owned, fixed-implementation `sin`/`cos`/`tan`/`exp`/`log`/`pow`/`atan2`/`hypot` family — documented accuracy (≤ 1 ulp target, per-function error tables published), identical bits on every certified target, locked by cross-platform vectors in CI — serving compiled evaluation, `#eval`, and `native_decide`-class computation, so a result computed on aarch64 replays on x86-64. Judgments never consume floats (the kernel treats `Float` operations as opaque), so the soundness surface is untouched; the evaluation surface gains cross-platform bit-stability the Reference cannot promise even to itself, recorded as a Behavior Note (the platforms disagree with *each other*; we pick one correct answer and keep it everywhere). No platform libm is linked, period.

## 7. Grimoire: environment & module codec (fln-env, fln-olean)

**7.1 The environment.** Semantically the Reference's: a map from names to constants (axioms, definitions, theorems, opaques, inductives/constructors/recursors, quotients), plus environment extensions with import-time replay of serialized entries, attribute state, and the module import graph. Mechanically ours: a persistent (HAMT-style) map with structural sharing, so snapshots are O(1) — the primitive under Athanor's speculative parallelism (§10.6), Lantern's per-request views (§14.3), and Envoy's search-tree states (§16.3). Extension replay preserves the Reference's ordering semantics exactly; mathlib's custom extensions are data in regions (§1.6) and round-trip without executing code; an extension payload understood only as opaque bytes is preserved losslessly, flagged in provenance, and honestly blocks fine-grained invalidation and cross-epoch migration through it rather than being guessed safe. Every environment extension declares its merge and checkpoint semantics in a registry, so branching and merging an environment includes its extensions *by contract*, not by each author's memory. Each environment commit carries a **logical root** (digest over declarations + extension deltas + options, excluding wall-clock, paths, and scheduler traces) and a separate operational-metadata root — two hosts producing the same trusted environment share a logical root even when their operational manifests differ; the logical root is the cache key the Ledger, receipts, and Envoy all speak.

**7.1b The term store: total interning.** Expressions, levels, and names are hash-consed **contractually, not opportunistically**: one node per full structural identity (binder names and `mdata` included, because metaprograms observe them and faithful `Expr` behavior demands exact round-trip), with pointer/ID equality *being* structural equality — `Expr.equal` collapses to a compare, and the kernel's caches key on identity pairs. An auxiliary **alpha-canonical digest** (binder names stripped) indexes kernel-side defeq/whnf caches so alpha-varied duplicates share reduction work without ever confusing the metaprogram-visible plane. Sharing is maximal and *measured*: every store keeps sharing statistics and enforces expanded-weight budgets, because a term small on the wire can be astronomical when unshared — an adversarial DAG-bomb is a first-class, typed, inspectable condition (§8.2c's resource algebra counts expanded weight, not bytes), and serialization preserves sharing exactly in both codecs.

**7.2 The olean codec, read.** Header (magic, version fields, base address) parsing; region relocation; `ModuleData` traversal into environment structures; validation levels from "trust the producer" (mmap-speed, the Reference's own posture) through "structural audit" (tag/bounds/reachability sweep, still fast) to "full recheck" (Crucible over every constant — G1's mode). The layout inventory is generated from the pin into `OLEAN_CONTRACT.md` exactly as §6.5 does for the ABI, with per-field provenance and a versioning note per Reference release in the supported epoch set.

**7.3 The olean codec, write.** Region construction that the Reference loads: identical header discipline, identical object layout, and — in `faithful` mode — **byte-identical output** for the round-trip law FL-INV-04 (read a Reference olean, rebuild the region, diff bytes; then the harder direction: elaborate a module fresh and match the Reference's emission for the deterministic subset, with any residual serialization freedom pinned by a registered CGSE policy). Byte-exact write is a G0 spike (G0-5) because it forces every layout truth to the surface early; it is also what makes FrankenLean a *transparent* participant in mixed toolchains and CI farms. Two identities travel with every stored object: a **semantic hash** (canonical meaning under the schema) and a **byte hash** (the stored encoding) — re-encoding and compression can be honest about changing bytes without pretending to change meaning. Cross-file artifact sets (`.olean` + `.ilean` + native siblings) promote through a transaction manifest: staged, verified, then published atomically together or rolled back together; a crash mid-promotion leaves unreachable staging, never a half-published module.

**7.3b The `.ilean` codec.** Editor-metadata files are an exact epoch projection: spans, identifier occurrences, and resolution targets with the pin's field order, omissions, and position conventions, generated from the same normative schema layer as the olean codec so writer and checker cannot skew. Richer server metadata lives beside it in Palimpsest; `.ilean` stays exactly as boring as the ecosystem expects.

**7.4 olean-next (frontier).** The second lane: a content-addressed, position-independent module format — per-declaration records keyed by Ledger hashes, structure-shared term DAGs with dictionary compression, mergeable and diffable, mmap-friendly without relocation, carrying certificate attachments (§8.6) inline. olean-next never replaces olean in `faithful`/`sound` artifacts; it is the Ledger's native interchange and the format proof-cache federation speaks. A `fln olean` tool converts both directions and prints human-readable diffs of module contents — the un-opaquing of weakness §1.11-8. olean-next capsules come in two shapes: **thin** (references into the shared CAS) and **sealed** (a complete transitive object pack, verifiable on an air-gapped machine — the archival and audit form). Attested evidence objects (release capsules, receipts, Parity-Ledger snapshots, benchmark baselines) may carry RaptorQ repair sidecars via asupersync's coder; repair reconstructs to staging, re-verifies the canonical hash and a full schema decode, publishes without changing identity, and journals the corruption/repair event — repair is a durability feature, never a trust shortcut.

**7.5 Import at daemon scale.** Import resolution builds one shared, immutable, reference-counted-as-persistent heap of loaded regions per toolchain epoch; every consumer (server sessions, build workers, MCP snapshots) maps the same pages. The ≈60 s/GBs-per-worker multiplication becomes ≈once-per-daemon; PG-4 quantifies the promise.

## 8. Crucible: the trusted kernel (fln-kernel, fln-bignum)

**8.1 The covenant.** `fln-kernel` is ≤ 12 KLOC of `forbid(unsafe_code)` Rust with no dependencies beyond `fln-bignum`, Grimoire's *read-only* NativeHeap term views, and `core`/`alloc` — zero I/O, zero threads, zero global mutable state, zero plugin hooks, a dependency closure that prints on one page and is *generated from the actual graph* into the TCB inventory every CI run. It exports exactly one authority: `check : Environment × Declaration → Verdict`. Nothing else in the program can admit a constant (FL-INV-02); the elaborator, the engines, the codec's fast paths, and every native façade implementation in the Mirror are untrusted by construction. The covenant is enforced mechanically: a CI check counts lines, walks the dependency graph, and fails the build if the kernel grows past the bound or acquires an import — growth requires amending this document first.

**8.1b The Judgment Specification comes first.** W3's first deliverable is not code: it is `KERNEL_CONTRACT.md` (Appendix A's normative expansion) — a rule-by-rule account of the implemented type theory, each rule anchored to the Reference source lines that define it and to the literature that justifies it, against which both engines are written and the Tribunal's term generators are built. The Reference's specification is its C++; FrankenLean's specification is a document the C++ is *checked against*, rule by rule, by the differential harness. That inversion is itself a product.

**8.2 The judgments.** The full inventory of §1.2, implemented rule-for-rule and frozen as Appendix A: typing for every expression constructor; universe-level normalization with the `imax` collapse; definitional equality with beta/eta(fun,struct)/delta-by-height/zeta/iota/proof-irrelevance/quotient/literal rules; inductive validation (strict positivity, universe constraints, mutual and nested via the Reference's nested-to-mutual compilation, reflexive types, K-like subsingleton elimination); structure eta and projection typing; `Nat`/`String` literal acceleration through fln-bignum for the same operation set the Reference accelerates (so `decide`-class proofs have the same asymptotics); recursor-rule application; and the environment-level rules. Divergence policy is absolute: a term the Reference kernel accepts and Crucible rejects (or vice versa) at the pin is a release-blocking defect, no matter which side is "mathematically nicer" — kernel disagreement is the one place `sound` mode has no license — with one asymmetric carve-out: if the differential harness establishes that the *Reference* accepts a term the type theory rejects (an upstream soundness bug — it has happened), FrankenLean matches the **theory**, files the reproducer upstream, and records the divergence in the Behavior Notes; logical soundness outranks bug-parity, and the affected row simply cannot be faithful-exact (a Reference-accepted/Crucible-rejected finding is therefore triaged first as an oracle-or-spec investigation, never auto-classified as our defect *or* theirs).

**8.2b Typestate admission.** The kernel accepts input only through a one-way gate: the decoder produces `UnvalidatedExpr` (typed indices, bounded lengths, no raw pointers); the scope validator produces `ScopedExpr`; only type checking produces `CheckedExpr` — constructors that could violate well-scopedness are private, and no integer is accepted as two semantic kinds (`DeclNameId`, `ExprNodeId`, `BinderDepth`, `ReductionFuel`, `EnvCommitId` are distinct newtypes). Submission is a `DeclarationEnvelope` (epoch, base logical root, name/kind, type, value, level params, safety/reducibility, source identity, requested resources), and the verdict is exactly one of `Accepted { receipt }`, `Rejected { stable_error_class, blame_slice, receipt }`, `Inconclusive { resource_reason, receipt }` (FL-INV-07: never conflated with rejection, never promoted to acceptance), or `InternalFault { crash_capsule }` — the last is process-fatal in certified profiles and quarantines every cache the faulted process touched until an independent process revalidates them.

**8.2c Resource totality.** A sound kernel that a small artifact can spin forever is operationally incomplete. Every judgment runs under a typed budget drawn from the caller's `Cx` — node count, depth, expanded weight, reduction steps, allocation — with cooperative cancellation checkpoints; exhaustion is a typed `Inconclusive` carrying the consumption profile, never a hang, a host-stack overflow (no recursive host traversal over attacker-controlled depth), or an OS kill; no error path pretty-prints an unbounded term. Heartbeat parity (§8.5) is the `faithful`-mode *projection* of this algebra onto the Reference's counting law.

**8.3 Dual engines, an independent checker, and consensus.** Engine K1 is the **certified evaluator**: a direct, boring, small-step-flavored implementation optimized for auditability — the code a skeptical logician reads in an afternoon. Engine K2 is the **NbE accelerator**: normalization-by-evaluation with environment-aware closures, memoized whnf keyed on (term-hash, transparency), and lazy delta with height comparison — the code that makes `decide`, big `Fin` computations, and mathlib's defeq-heavy corners fast. Both engines implement the same judgments; K2's verdicts are spot-audited against K1 continuously (sampled per run, exhaustively in Tribunal soaks), and any disagreement fails loudly and ships as a fixture. `--kernel k1` forces the boring path for audit builds. The size covenant counts both. Every fast-lane shortcut in K2 carries exactly one of three justifications, registered per D7: a local proof argument in the kernel design note; a certificate checked by a simpler routine; or differential evidence — which licenses an optimization and never impersonates a proof.

**8.3b The independent checker (`fln-checker`).** A deliberately *different* second implementation, not a feature flag over `fln-kernel`: separate normalization algorithm (eager where K1/K2 are lazy), simpler persistent maps and substitutions, its own decoder over Grimoire's canonical wire objects (never the primary's heap nodes), no dependency on the unsafe membrane, an independently derived error-class mapping, a separate build artifact, and a WASM build for a foreign execution substrate. Sharing data schemas is acceptable; sharing semantic implementation is not, and CI proves the non-sharing rules structurally.

**8.3c Judgments as objects; consensus as policy.** Every kernel decision materializes as a canonical, hashable `Judgment { input_digest, env_logical_root, verdict, engine_id, fuel_profile }` — comparable because B4's determinism makes it so. Verification policies, versioned and journaled into every receipt: `standard` (K1/K2 on every declaration; `fln-checker` on a deterministic content-seeded sample plus all high-risk forms — never runtime randomness); `release` (checker on the full release closure); `paranoid` (both lanes + checker everywhere, optionally on distinct hosts); `compat-bench` (sampled, benchmarking only, never attestable). **Disagreement halts.** A divergence between any two engines blocks publication with both traces attached and the reproducing declaration auto-shrunk by the Tribunal's delta-debugger; it is diagnosed, never outvoted — there is no quorum that can vote a wrong judgment through, and the certified engine's seat is mandatory on attested paths. The economics are honest: the checker roughly doubles kernel CPU where it runs, and the kernel is a minority of end-to-end elaboration wall-clock, so the attested-path tax is single-digit percent (PG-1b measures it) — the cheapest unit of assurance in the whole stack. What consensus is *not*: N-version theater (the engines share the Judgment Specification, so correlated spec bugs are possible — which is why §8.7's *foreign* witnesses exist and why the spec is a document), and not a proof of the metatheory (Appendix A states exactly what is and is not established).

**8.4 fln-bignum.** Kernel-grade arbitrary precision: limb-based magnitude representation matching the ABI's bignum-object layout (so Marrow's mpz-shim is a view, not a conversion), schoolbook/Karatsuba/Toom multiplication with thresholds tuned on the kernel-reduction corpus (not on synthetic benchmarks), Knuth-D division, binary/Lehmer gcd, exact decimal parsing/printing for literals, and constant-time-ness explicitly *not* claimed (this is a prover, not a KMS). PG-K gates it against GMP on the corpus's actual operation mix; the honest expectation is parity on small operands (which dominate) and a bounded gap on giants, with the gap published, not hidden.

**8.5 Fuel parity.** The heartbeat law (§4.2) is counted where the Reference counts it, so `faithful`-mode timeout behavior is reproducible to the tick; `maxRecDepth` mirrors the Reference's recursion accounting. Determinism of *failure* is part of determinism. In `sound` mode the same budgets surface as the typed resource algebra of §8.2c; `sound` may finish earlier than an upstream timeout, never later-with-different-success (BN-08).

**8.6 Receipts.** Every checked declaration can emit a **certificate**: the compressed, structure-shared elaborated term DAG plus the reduction hints that make independent rechecking cheap (unfold order, literal-arithmetic transcripts for the accelerated ops so a tiny verifier can replay them without its own bignum trust). Certificates attach to olean-next records, feed the export tool (`fln export`, speaking the community's established export format for cross-prover checkers), and power the headline product of G1: **the Independent Judge** — `fln check-olean` over the entire mathlib olean set, in minutes, on a kernel that shares no code, no compiler, and no bignum with the thing it is checking. Two implementations agreeing is evidence; two *foreign* implementations agreeing is the strongest evidence this ecosystem can currently buy. Certificates obey one law: the certificate checker must be **simpler than recomputation**, must reject unknown versions, and must fall back to recomputation on any failure — a certificate is an accelerator, never a wider TCB. Receipts additionally bind the kernel and checker *build hashes*, the policy, the fuel profiles, and both verdicts, and may ship a **warm defeq-cache cartridge**: the memoized reduction history that makes third-party rechecking verification rather than search. Attested receipts append to a **transparency log** (Merkle Mountain Range over receipt roots, signed checkpoints): a third party verifies inclusion and consistency without trusting the logger, and equivocation is detectable — "checked" stops being a verb someone had to witness and becomes an artifact anyone can audit. And because K1 is specification-first, `forbid(unsafe)`, and WASM-clean, the **WASM Judge** falls out: the certified engine compiled to a single WASM artifact that re-checks sealed capsules against the same vectors with the same bits, in a browser tab — zero-install verification as a public good (§16.4b).

**8.7 The differential posture.** Standing Tribunal rigs: Crucible vs. the Reference kernel (via lean4checker on shared oleans); Crucible vs. lean4lean's verified rules (fixture translation); Crucible K1 vs. K2 (continuous); fuzzed-term storms (well-typed generators + mutation) where *any* pairwise disagreement is a finding. The kernel is the one subsystem where we volunteer to be outnumbered — and the external constellation (lean4checker on shared oleans; lean4lean-class verified rules and export-format checkers via `fln export`) can join the §8.3c policy as *optional consensus members* on sampled classes from day one: full-corpus external witnessing is a throughput project, but sampled foreign witnessing is cheap and even one caught divergence pays for the wire.

## 9. Quill: the parser & macro engine (fln-parse, fln-syntax)

**9.1 The engine.** Token tables and Pratt-precedence parsing with the Reference's category/leading/trailing model; `Syntax` trees with identical node kinds, atom/ident/node structure, and SourceInfo discipline (original vs. synthetic, whitespace attachment, byte positions in UTF-8 exactly as upstream computes them — positions are wire-visible through diagnostics and LSP, so they are conformance surface, not detail). The extensibility contract is preserved in full: parsing interleaves with elaboration, commands install syntax/macros/elab rules visible immediately, and precedence interactions match. Internally, the source lives as a lossless rope with a persistent green tree (exact bytes, trivia, comments, UTF-8/UTF-16 positions) whose red view adds context pointers per document snapshot; every node carries an **origin set** — literal span, macro-definition span, macro-call span, quotation/antiquotation spans, recovered markers — projected to the Reference's single-primary-span convention at the wire while the full provenance graph feeds Palimpsest, so "edit the theorem call" and "edit the macro definition" are distinguishable code actions. The grammar state is versioned data, not hidden globals: a **GrammarEpoch** content-hashes the active syntax declarations, parser descriptors, precedence tables, scoped notation, token parsers, and relevant options; every parse node binds the epoch it was recognized under, memoization keys on (position, category, precedence, GrammarEpoch), and a syntax-changing command invalidates exactly the parse products that read the changed components. Parser registrations emit typed effect summaries (`AddsSyntax`, `AddsToken`, `ChangesPrecedence`, `OpensScope`, `RegistersMacro`, `OpaqueParserEffect`) — an opaque effect is legal for fidelity but forces the conservative suffix barrier and disables distributed parsing past that point.

**9.2 Hygiene.** Macro scopes as the Reference implements them (scope-decorated names, `Syntax` quotation/antiquotation semantics, `macro_rules` matching), verified by a dedicated fidelity rig: corpus files whose meaning depends on hygiene corner cases, parsed and elaborated under both implementations with environment diffs (G0-4). Hygiene is the classic reimplementation graveyard; it gets spike status, not faith. Generated identifiers derive from stable expansion paths (origin command → macro invocation → quotation path → local ordinal), never from thread timing or allocator addresses — determinism of generated names is a compatibility requirement, since metaprograms and diagnostics observe them. Macro expansion is a **transaction**: an invocation receives syntax + hygiene seed + grammar/extension snapshot + declared capabilities + a deterministic budget, and returns expanded syntax, generated names, messages, a source map, extension reads, and capability events; it is memoizable exactly when all reads are known — a file-reading macro caches against the capability-recorded canonical path *and content hash*, a clock-reading macro is opaque in `faithful` and denied by default in hardened `sound` profiles.

**9.3 What we improve (sound mode).** Error recovery that survives common malformations to keep serving diagnostics/completions downstream of an error (the Reference's recovery is uneven; ours is specified: recovery productions per category, resynchronization tokens, and a rule that recovery may add diagnostics but never change acceptance); incremental *re-lexing and reparse* below command granularity feeding Lantern's snapshots; and parse-tree provenance into Palimpsest so "where did this syntax come from" (macro expansion chains) is a query, not an archaeology. Two boundary products serve the editor: the **exact command stream** (the Reference parser's semantics, sole authority over meaning) and a **speculative boundary map** (a conservative scanner that recovers enough structure after malformed edits to schedule work and show syntax diagnostics); speculative boundaries can never commit declarations, and disagreement is journaled as a recovery event. One law rules recovery: no recovered node reaches Athanor without an explicit `RecoveredSyntax` marker, and no declaration dependent on an unacknowledged repair is ever published.

## 10. Athanor: the elaborator (fln-elab; Synod within)

**10.1 The tower.** The monadic stack's *semantics* — Core/Meta/TermElab/Tactic state shapes, metavariable contexts with delayed assignments, local contexts, backtracking discipline, option threading, message logs, InfoTree recording — implemented natively as NativeHeap structures, with Mirror-façade views projected into Marrow objects exactly where user metaprograms touch them (§4.3, §6.1b): the toolchain pays the ABI shape only at the seam it actually crosses. Elaboration state is explicit, not ambient: each command runs in an `ElabTxn` (base logical root, local context, metavariable store, universe store, constraint queue, extension overlay, message buffer, InfoTree builder, capability context, decision ledger, deterministic seed, budget), and nested elaboration uses child transactions with exactly five outcomes — commit all products; commit diagnostics/info only; roll back completely; expose a candidate plus unresolved obligations; or fork into deterministic alternatives for overload/tactic search. Transactional rollback is *audited* by the Tribunal (leak tests prove no failed alternative leaves assigned metavariables, registered declarations, or messages behind) — the bug class upstream's ambient monads make chronic is made structural here. Metavariables form an explicit dependency graph (type, context, kind, origin, assignment state and justification, delayed constraints, closure identity) with targeted wake-up: assigning one reschedules only constraints that actually read it — the substrate of both scheduler precision (§10.6) and the "why is this goal stuck" answer (§15.2).

**10.2 Unification & defeq.** Transparency-aware `whnf`/`isDefEq` mirroring the Reference's approximation ladder: syntactic fast paths, first-order approximation, Miller patterns, delayed assignments, postponement, unfolding by reducibility and height, structure-eta awareness. This is the subsystem where "bug-for-bug" earns its keep: mathlib *depends* on the precise power of the unifier (proofs elaborate because a particular approximation fires), so the Tribunal's T3 rig diffs unifier outcomes across the Corpus and every divergence is triaged as fidelity-defect vs. Behavior-Note candidate — with the strong prior that it is a defect. The defeq *tail* (the pathological problems that dominate wall time) is G0-7's spike: the twenty nastiest Corpus defeq problems, timed on K2's memoized engine, because if the tail doesn't fall, PG-2 doesn't happen. Under the Oracle-Only Law the fidelity burden here is entirely ours — there is no hosted upstream unifier to fall back to — so the mitigation is industrial: the **instrumented oracle** (§18.3) mines golden decision traces (which approximation fired, what postponed, what unfolded, in what order) from the Reference at Corpus scale, and the unifier is implemented against those traces, not against vibes. Failures are recorded as typed **constraint cells** (`HasType`, `IsDefEq`, `SynthesizeInstance`, `Coerce`, `ResolveName`, `ElaboratePattern`, `ProveTermination`) with origins, dependencies, attempts, and outcomes, from which Palimpsest computes **blame slices** — the minimal causal subgraph explaining a contradiction or a stuck goal (§15.2); `faithful` renders the Reference's diagnostic text, `sound` renders the slice.

**10.3 Term elaboration.** Coercions, implicit/strict-implicit/instance-implicit binder handling, auto-bound implicits, structure instances and anonymous constructors, `match` compilation into auxiliary matchers with equation lemmas and splitters matching the Reference's generated-name and shape conventions (metaprograms pattern-match on these artifacts; they are API), `deriving` handler dispatch, attributes, `variable`/section/namespace resolution with the Reference's ambiguity rules, and the recursion elaborators (structural, well-founded with the same measure-synthesis surface, `partial`, `unsafe`) lowering to exactly the kernel-visible constructions of §1.2.

**10.4 Synod: instance resolution.** The Reference's semantics — priorities, `outParam`/`semiOutParam`, default instances, tabling/caching behavior, the result the ecosystem's diamonds depend on — behind a rebuilt engine: instance search compiled to an indexed relational form (discrimination-automaton candidate retrieval, subsumption-checked tabling, cycle detection with franken_networkx diagnostics), with **provenance recorded per resolution**: every attempted instance, why it failed (unification residue, priority loss, depth), as a Palimpsest tree. `sound` mode's "why is there no instance" answer (§15.3) is this engine's exhaust, not a second implementation. Faithful mode pins search order to the Reference's; sound mode may reorder *only within proven-confluent regions* (same result, better time), CGSE-registered. Every search emits a `SearchCard` (explored candidates, priorities, unification residues, cycle decisions, cache hits, selection) — provenance and negative-cache key in one artifact, with negative entries valid only under their exact dependency key (instance-table identity, transparency, options, local-context fingerprint). Name resolution, coercion graphs, and discrimination indexes are immutable content-addressed derivations of the environment commit, incrementally maintained by deltas and cached as Ledger artifacts — faithful iteration order preserved even where the internal structure is faster. Termination elaboration additionally records a machine-readable `TerminationCard` (call graph, candidate measures, generated obligations, chosen theorem dependencies) — the artifact Bloodhound's repair lane consumes when a termination proof breaks (§16.1).

**10.5 Tactic execution.** The tactic *framework* is native: goal management, metavariable assignment, `evalTactic` dispatch, `elab_rules`/`macro_rules` registration and matching, focusing/structuring combinators, `evalExpr`-class services, and the InfoTree hooks are Athanor code, and the core tactic set (`rfl`-class closing, `intro`/`apply`/`cases`/`induction` machinery, `simp` via Anvil, `decide` via Crucible, `omega`/`bv_decide` via Verdict, `norm_num`-class evaluation) is native behind its Mirror façade rows — each with equivalence class, golden-trace fixtures, and its own Parity-Ledger L-level. *User* tactics — mathlib's, aesop's, a project's `elab` blocks — run on Golem and call that native framework through the façade, receiving `GoalState` views and capability contexts; no tactic, native or user, can construct a trusted declaration except through Crucible (FL-INV-02). The framework's observable semantics (state threading, backtracking, error propagation, InfoTree shape) are conformance surface, proven by the metaprogram-corpus rig of §18.2 running the ecosystem's real tactic code.

**10.6 The scheduler (Bet B4).** Declarations become dataflow nodes: parse yields a conservative dependency pre-scan; elaboration proceeds speculatively in parallel regions (asupersync) against O(1) environment snapshots (§7.1); a declaration that turns out to read a not-yet-committed name suspends on its true dependency or aborts-and-retries; commits merge in **canonical order** (source order per module; module order per the Ledger's topological policy) so the resulting environment, diagnostics, and InfoTrees are those of the sequential elaboration — bit-for-bit, at any thread count (FL-INV-01). Side-effecting commands (attribute mutations, `initialize`, option changes, syntax installation) are ordering barriers detected conservatively: unknown effects serialize, known-commutative effects (the registered set) parallelize. This is the discipline upstream's `Elab.async` lacks and the CGSE doctrine supplies: parallelism with a determinism *proof obligation*, discharged by construction and re-verified by PG-5's thread-matrix every commit. Speculation extends across modules through the Ledger (§13.2): downstream declarations elaborate against upstream *hashes-in-flight*, with mis-speculation costing a retry, never a wrong answer. Stated as optimistic concurrency: transactions get source-order IDs; eligible work executes in parallel against immutable snapshots; commit-time read-set validation against the latest committed predecessor state either commits the delta in canonical order or discards/rebases — there are no last-writer-wins rules, and conflicting extension mutations serialize. Command **effect summaries** make eligibility precise: each command's reads and writes are captured dynamically as typed effects (`ReadsDecl(name, Type|Value|Attributes)`, `ReadsInstances(head)`, `ReadsSimpSet`, `ReadsGrammar`, `ReadsOption`, `WritesDecl`, `WritesInstance`, `WritesGrammar`, `WritesEnvExtension`, `UsesCapability`, `Opaque(reason)`), and the Tribunal validates them **by perturbation**: mutate an allegedly-unread input and require the product unchanged — a failing perturbation demotes the producer to `Opaque` (Reference-style suffix barrier) and opens a Parity-Ledger bead. Precision is earned by evidence, never assumed; the system starts conservative and ratchets.

## 11. Golem: the metaprogram compiler & VM (fln-comp, fln-vm)

Golem exists because Lean is a programming language: user metaprograms — mathlib's tactics, project macros, lakefiles, `#eval`, compiled executables and plugins — are Lean code and must run somewhere. Golem compiles and executes them with the Reference's observable semantics against the Mirror's native builtin surface. It is emphatically *not* a host for upstream's implementation: the elaborator, tactic framework, and compiler never ride it (D8); the only Lean it ever executes is user-class code and source-elaborated library code (§4.3).

**11.1 The pipeline.** Elaborated terms → **FIR** (FrankenLean IR: an ANF/LCNF-class form with the Reference-established disciplines: specialization of instance/closed arguments, simplification, closure conversion, explicit boxing, and reference-count insertion with borrow inference and constructor-reuse analysis — kept because FBIP-style reuse is the performance model the ecosystem's code is written against) → **FLBC**, a register bytecode whose values *are* Marrow ABI objects. That last identity is the design's quiet masterstroke: because interpreter values are ABI values, calls cross between interpreted code, Iron-compiled code, Reference-built plugins, and the Mirror's native façade with zero conversion — one calling convention for the whole membrane. Every pipeline stage has a versioned IR schema, a deterministic printer, and a validator; every optimization pass is a pure `ValidatedIr → (ValidatedIr, PassWitness)` transformation with declared preconditions, a preservation argument or executable validator, a resource-growth bound, deterministic ordering, and a fallback to its input — the RC-insertion pass in particular runs an ownership validator (no use-after-consume, balanced joins and exception edges, capture correctness, ABI ownership contracts) that is deliberately simpler than the optimizer it audits. Executable-closure selection records *why* each declaration is retained (direct call, closure capture, initializer, reflection, exported symbol, conservative opaque edge), so dead-code questions have answers.

**11.2 The interpreter.** Threaded dispatch, inline caches on constructor tags and closure arities, direct-call fast paths for the intrinsic table, allocation through Marrow (so heartbeats count identically), and the Reference's semantics for panics, `partial` non-termination, stack depth, and `checkSystem` interruption polls. PG-7 gates it at ≥ 3× the Reference interpreter on the tactic micro-corpus — a deliberately conservative floor chosen because the interpreter is the default vehicle for the long tail of user tactics.

**11.3 Iron-JIT (frontier, standard-only).** A baseline template JIT from FLBC to x86-64/aarch64 in `fln-unsafe-jit`: no IR heroics, just call-threaded code with the inline caches burned in — the tier that makes hot tactic loops approach precompiled speed without a C compiler or an external backend. Certified builds never enable it (interpreter results are the definition); its results are bit-identical to the interpreter's by construction and spot-checked by a standing dual-run rig.

**11.4 The intrinsic table is the Mirror's floor.** The Reference's `@[extern]` census (Appendix C: every extern symbol in `Init`/`Std`/`Lean` at the pin, extracted mechanically) implemented as native Rust intrinsics — Nat/Int/Array/ByteArray/String/Float ops, IO, Task, ST, environment access, compiler-API entry points — beneath the broader toolchain-API façade of §4.3, which additionally serves the Lean-defined machinery (`Lean.Meta.*`, `Lean.Elab.*`, the tactic framework) natively. Float arithmetic is IEEE-754 binary64 with the Reference's printing/parsing; transcendentals route through fln-libm (§6.8); `UInt*`/`USize` follow the ABI's width rules. Every row carries fixtures and an L-level; the census diff against a new Reference epoch is the first artifact of every pin advance.

**11.5 Initialization & effects.** `initialize`/`builtin_initialize` ordering, import-time extension replay interleaving, and `IO`'s real-world effects behave as upstream — under asupersync capabilities, so a `--pure-elab` audit mode can *deny* ambient filesystem/network/clock to metaprograms and report exactly which declarations demanded them (capability-scoped metaprogramming: a `sound`-mode observability win and a `frontier`-mode sandbox, never a semantic change to compliant code).

**11.6 `native_decide`: the wound, staged toward healed.** Supported with the Reference's semantics and the Reference's *caveat*, stated louder: it extends trust to Golem/Iron. Provenance marks every `native_decide`-dependent theorem; `fln audit --tcb` lists a project's trust surface (axioms, native_decide sites, plugins, façade classes), and `fln why-trusts <decl>` walks the transitive cone (§15.6). Then the staged repair, honestly tiered and never gate-blocking: **Tier 1** (sound-mode default at G4): every reflection-class evaluation emits a replay receipt — function-entry digests over the registered reflection set, branch decisions, literal-arithmetic transcripts, recorded against term identities, replayable by Crucible at a stated cost multiple; certified profiles verify always, standard samples by policy. **Tier 2** (G6, certified paths): trace-guided kernel replay becomes the default — `ofReduceBool` is *discharged*, not trusted, where the tier holds. **Tier 3** (research, stated as research): a verified evaluator proven against the Judgment Specification. Each tier stands alone; the trust ledger records which tier each build achieved, because "we no longer trust the compiler" is a claim with a scope.

## 12. Anvil: rewrite & decision engines (fln-anvil, fln-verdict)

Anvil's constitution is FL-INV-06: every engine is untrusted; every output enters the environment only through a kernel-checked artifact. That single rule buys unlimited freedom below it.

**12.1 simp, faithful core.** The Reference's simp semantics — simp-set discipline, discrimination-tree candidate retrieval, congruence closure interplay, `dsimp`/`simp only` behaviors, attribute-driven extension — as the compatibility baseline, implemented **natively end to end** (the simp driver, congruence interplay, and extension dispatch are Anvil code behind their Mirror façade rows, `obs` class), with the Corpus's simp-heavy files plus instrumented-oracle rewrite traces (§18.3) as the fidelity corpus. Mathlib's simp *extensions* and simprocs are user code on Golem, calling the native driver through the façade.

**12.2 Compiled rewrite indexes.** Per-library discrimination automata compiled at Ledger commit time and shipped as CAS artifacts (an olean-next attachment): mathlib's tens of thousands of simp lemmas become a preloaded matcher, not a per-session index build. Index staleness is impossible by construction (content-addressed against the simp-set's own hash).

**12.3 The e-graph lane (frontier).** Equality saturation over simp-set subsets with proof extraction: rewrites recorded as justifications, extracted proofs replayed through the ordinary lemma-application path, kernel-checked like everything else. Honestly scoped: conditional rewrites and binder-crossing rules restrict the eligible subset; the lane is an accelerator with a completeness fallback to §12.1, never a replacement. Its win condition is measured (PG-A profile), not assumed.

**12.4 Arithmetic & normalization.** `norm_num`-class evaluation and `omega` with native cores (fln-bignum-backed), certificate-producing, kernel-replayed.

**12.5 Verdict (fln-verdict).** An owned CDCL SAT solver — watched literals, VSIDS/CHB-class heuristics, clause management, inprocessing kept modest and honest — with **owned proof logging and an owned checker**: the `bv_decide` pipeline bit-blasts through Verdict, Verdict emits a checkable unsatisfiability certificate, and a small independent checker (kernel-adjacent in audit rigor, separate in code) validates it before the reflected proof is admitted. The external-binary TCB of §1.10 dies here. Parity posture: PG-9 tracks Verdict against the Reference's solver on the `bv_decide` corpus with a published gap budget; correctness is absolute from day one, competitiveness is a ratchet.

**12.6 grind posture.** `grind` is core-toolchain machinery at the pin, so under the Oracle-Only Law it is **reimplemented natively**: congruence closure, e-matching, and its theory hooks as Anvil engines behind façade rows, developed against instrumented-oracle decision traces and the Reference-as-oracle differentials, and staged deliberately — its Parity-Ledger rows enter at L1/L2 and ratchet to L3 on the grind corpus by G4 rather than pretending day-one exactness, because grind's semantics are still moving upstream and the census pins our coupling per epoch. Mathlib-layer automation (`aesop`, `polyrith`, …) is user code on Golem and needs no reimplementation at all. New engine lanes here (and in §12.3) promote by **shadow-run**: both lanes run on the same demand nodes, products diffed, divergence and latency distributions recorded, promotion per workload/epoch/platform cell with deterministic sampling continuing afterward and auto-demotion on any soundness, fidelity, or tail-latency breach — promotion is a matrix, never a global flag.

**12.7 Portfolios (frontier).** Speculative tactic racing (`first`-with-work-stealing, deterministic winner selection by registered policy, losers cancelled through their regions) and snapshot-cheap `try` trees — the search-side dividend of O(1) environment snapshots, exposed to users as tactics and to agents through Envoy.

## 13. Ledger: the build fabric (fln-ledger, fln-lake)

**13.1 The model.** Every declaration commit produces a **Ledger record**: `decl_hash = H(name, elaborated statement, elaborated value/body class, direct dep hashes, options-that-matter, toolchain epoch, mode)`, stored content-addressed (frankensqlite metadata + CAS blobs), with the dependency edges as first-class rows. The environment at any point is a Merkle frontier keyed by logical roots (§7.1); a module is a view; a build is a query. The record generalizes to a **demand node** — kind, implementation version, epoch, semantic inputs, declared reads/effects (§10.6's vocabulary), capability inputs, output schema, determinism class (D0–D4) — spanning five interlocking planes with separate retention: source, language, trust (kernel/checker receipts), build (facets, codegen, links), and service (LSP products, indexes). An LSP hover can be evicted without touching the checked declaration beneath it; retention tiers run **T0** trusted roots (durable, never sole-copy-evicted) → **T1** expensive reusable → **T2** interactive → **T3** speculative → **T4** observational (sampled), with eviction weighing recomputation cost, fan-out, and pressure.

**13.2 Incremental law.** An edit invalidates exactly the records whose hash inputs changed — the *true dependency cone*, computed on the recorded edge set (franken_networkx dominator/reachability machinery), not the import-file cone. Whitespace, comments, docstrings (for non-doc facets), reordering of independent declarations, and edits to unrelated declarations in an imported file invalidate nothing downstream. FL-INV-05 (cache soundness) is enforced by the conservative rule set — anything not proven irrelevant is relevant — plus a standing Tribunal rig that randomly recomputes hits and diffs. Cross-module speculation (§10.6) rides the same records. The Reference's `module`-system boundaries, where authored, are honored as *additional* coarsening the user asked for, never needed as the mechanism. Two refinements make the law sharp. **Early cutoff:** a re-run node whose output digest is unchanged *stops the cascade* even though its inputs changed — the salsa/Adapton discipline, with statement-hash vs. value-hash separation doing the work (an opaque proof body changing invalidates the proof's receipt and codegen closure, not its typed dependents). **The invalidation matrix**, published and fixture-tested per change class — trivia-only edit → source maps and positions; proof-body edit (type unchanged) → that declaration's check + receipt + executable closure; definition-body edit → value-readers/unfolders + codegen cone; type edit → typed dependents; instance add/remove → affected class-head search keys (the negative-cache dependency keys of §10.4); notation/syntax change → parse products under the affected GrammarEpoch components after the activation point; option change → nodes that declared that option read; plugin change → its classified reads plus ABI load products — each with its conservative fallback, and each *perturbation-validated* (§10.6): mutate an allegedly-irrelevant input, require bit-identical products, demote to opaque on failure. The distinction between **answer identity** and **trace identity** is explicit: a cached answer can stay valid while its performance trace would differ; faithful-diagnostic rows may demand trace-equivalent replay where search order is externally visible.

**13.3 The Lake surface.** `lake` CLI-compatible: `lakefile.lean` executes on Golem (the configuration DSL is just Lean — user code, so Golem executing it is the compatibility mechanism), `lakefile.toml` takes the fast path, targets/facets map onto Ledger queries, `require` fetches via D2's git tool, `lean-toolchain` and elan layout honored so `elan` multiplexes FrankenLean toolchains transparently. Exit codes, `--json`, and the query surface match the pin; the build *behavior* upgrade (decl-granular no-op detection) is `sound`-mode and Behavior-Noted, with `--faithful-invalidation` available for byte-for-byte CI reproduction studies. Internally two rebuild decisions are computed every time: the **Reference decision** (what the pin's Lake would rebuild — faithful mode never omits an output upstream would produce, though it may satisfy it from a valid Ledger record and still project the expected files/traces) and the **native decision** (which demand nodes actually changed); `fln build explain <target>` shows both, the delta, the changed input identities, any opaque barriers, and cache outcomes — "why did this rebuild" becomes an answer, not an inference. Dependency resolution is transactional (a failed fetch/validation can never leave a half-rewritten manifest presented as complete) and emits a resolution receipt; a project's **hermeticity level** is reported honestly on the ladder **H0** ambient-recorded → **H1** inputs discovered and hashed → **H2** capabilities declared, undeclared effects denied → **H3** fully content-pinned, same-platform reproducible → **H4** air-gapped sealed rebuild — H0 suffices for faithful compatibility; release claims state the level actually demonstrated.

**13.4 Artifacts.** olean emission per §7.3 (so downstream Reference toolchains consume Ledger builds), olean-next alongside in frontier lanes, Iron/C-backend object products for executables, and Folio doc facets. `--reproducible` builds emit the sidecar closure manifest.

**13.5 Cache federation.** The CAS syncs over atp (multi-donor, fountain-coded, resumable) or HTTPS fallback, entries signed and verified; `lake exe cache get`'s muscle memory is preserved by a `fln cache` shim that hydrates the CAS from the community's existing archives during the transition. Because records are keyed semantically, a cache populated by *any* machine on the epoch serves every project shape — the "first mathlib build on a laptop" experience becomes: sync frontier, elaborate your delta. A remote entry is untrusted data until validated locally: node key and schema, content hashes and closure, epoch/platform compatibility, capability-input binding, kernel/checker receipts where relevant, signature policy, and decompression bounds — a malformed entry becomes a miss plus a security event, never trusted-state poison; concurrent fill is lease-suppressed, and corruption quarantines are first-class records.

**13.6 Epoch-stable caching.** Toolchain bumps re-key `toolchain epoch` — but the Ledger records *why*: a pin advance ships with a machine-diffed census (ABI/olean/extern/option deltas), and records whose inputs are provably untouched by the delta may carry **epoch bridges** (frontier, off by default, conservative allowlist) so the monthly treadmill stops meaning "the world from zero." This is the one place we out-ambition upstream's own roadmap deliberately, and it is gated behind its own soundness rig. The pin advance itself is the mechanical **epoch ratchet** of §22.5; alongside it ships a **migration autopilot** — audit a project against the new epoch's diffed census, propose source/config rewrites from the repair classes of §16.1, run lockstep builds, and emit an apply-transaction with a rollback bundle; it never edits a project without an explicit apply.

**13.7 Watch & distribute.** `lake build --watch` (filesystem events → cone recompute → live frankentui progress) and distributed elaboration (remote workers pulling Ledger tasks over asupersync RPC, results re-verified by local Crucible on arrival — distribution without distributed trust).

## 14. Lantern: the server (fln-server)

**14.1 The wire.** LSP with the `$/lean/*` dialect: `plainGoal`, `plainTermGoal`, the `rpc/*` session protocol (connect/call/release, reference-holding semantics), `waitForDiagnostics`, semantic tokens with the Reference's legend, diagnostics shapes, and the widget delivery protocol — byte-compatible, so `vscode-lean4` (and every other client in the ecosystem) connects to `lean --server` and cannot tell. Transport is asupersync's plumbing; the watchdog/worker *process* topology is not replicated — only its observable behavior is (per-file fault isolation becomes per-session region isolation: a crashing elaboration cancels a region, not a process, and the daemon's supervisor restarts the session with the same client-visible semantics as a worker restart).

**14.2 One daemon, one heap.** All open files, all projects on a toolchain epoch, one process, one shared immutable import heap (§7.5). The arithmetic this deletes is the ecosystem's most-cited pain: today's ≈60 s × N workers × GBs × N becomes once, shared, mapped. PG-4 and PG-6 pin the claim.

**14.3 Snapshots & incrementality.** Command-granular snapshots as upstream, extended downward: Quill's sub-command reparse plus Athanor's dataflow means an edit inside a proof re-elaborates from the finest enclosing checkpoint (tactic-block granularity where the Reference's incrementality already reaches; declaration granularity as the floor everywhere else), against O(1) environment snapshots, cancel-correct under typing bursts (stale requests cancel through their regions; FL-INV-01 makes the race harmless). The InfoTree is produced identically for the wire and *additionally* indexed for Palimpsest queries. Every open document version maps to an immutable `DocumentSnapshot` (uri, version, content id, project logical root, GrammarEpochs at boundaries, command transactions, **publication generation**); edits derive a new snapshot, and superseded requests keep their snapshot but lose publication authority — the stale-diagnostics-into-a-new-buffer race is structurally impossible, and every outward message rechecks its generation first. Diagnostics deduplicate by **stable identity** (typed cause + primary origin), not string equality, so position-only changes update in place instead of flickering; categories separate lexical / macro / elaboration / kernel / capability / build / divergence / advisory, with faithful mode emitting only epoch-expected categories and text. Code actions carry idempotency keys and preconditions (document, version, range, goal identity); applying one re-validates the precondition, stages the edit, re-elaborates, and reports — an action produced against a stale goal is rejected with a structured stale-state response, never applied optimistically. Clients may negotiate `$/frankenlean/progressiveProofState` — `ParseReady → ProvisionalGoals → VerifiedGoals → ImpactComplete`, same request identity, monotonically stronger grade, provisional never cached as trusted — while non-negotiating clients see exactly the Reference sequence. Multi-project sessions bind each root to its epoch capsule: a v4.32 file never elaborates in a v4.33 environment because two editors are open. And every session can emit a **replay bundle** (init transcript, edit and request sequence, content ids or sealed sources, scheduling profile, normalized outbound transcript): `fln server replay bundle` reproduces the session and reports the first divergence — server bugs become deterministic objects.

**14.4 The session API for machines.** The same snapshot machinery exposed as stable handles: fork a proof state, apply a tactic, read goals, diff environments — O(1) per branch, thousands of branches per daemon. This is the substrate Envoy (§16.3) sells; the DSP-pipeline papers' 18–735 s per-branch reconstruction cost is the number this section exists to embarrass.

## 15. Palimpsest: provenance, the causal proof graph & diagnostics (fln-trace)

**15.1 The substrate.** Structured elaboration events — unifier attempts, instance candidates, macro expansions, scheduler decisions, cache hits, kernel verdicts — recorded per declaration into ring-buffered traces (always-on cheap tier) and full archives (opt-in, frankensqlite-backed), keyed to Syntax ranges. Recording never changes results (observer discipline; enforced by dual-run rigs). From the journal, Palimpsest maintains the **causal proof graph** (Bet B7): typed nodes (source ranges, commands, syntax/macro expansions, declarations, instances, searches, tactic steps, kernel/checker verdicts, extension entries, build facets, capability sessions, branches/merges) and — the part flat dependency lists cannot express — **typed edge families**: `IMPORTS`; `READS_TYPE` vs `READS_VALUE` vs `UNFOLDS`; `SELECTS_INSTANCE`/`REJECTS_INSTANCE`; `USES_SIMP_RULE`; `USES_COERCION`; `EXPANDS_FROM`/`GENERATED_BY`; `READS_GRAMMAR`/`WRITES_GRAMMAR`; `PROVES`/`CHECKED_BY`/`CROSS_CHECKED_BY`; `COMPILES_TO`/`EXPORTS_SYMBOL`/`INITIALIZES_BEFORE`; `READS_FILE`/`SPAWNS_TOOL`/`LOADS_LIBRARY`; `INVALIDATES`/`REUSES`; `SUGGESTED_BY`/`ACCEPTED_FROM_SUGGESTION`; `MIGRATES_TO`/`REPAIRS`. Every edge carries span, phase, policy/epoch, receipt id — and a **completeness class**: `Complete` (all relevant reads captured *and perturbation-tested*), `Conservative` (may over-approximate, must not miss), `Observed` (one execution path; alternatives may exist), `Opaque` (introspection blocked), `Inferred` (heuristic). The welding law, stated once and enforced everywhere: **only Complete and Conservative edges may drive Ledger invalidation or any trust decision**; Observed/Inferred edges may rank, explain, and suggest, never invalidate — provenance and cache soundness are the same discipline or they are neither (FL-INV-05). From an edit or delta the graph computes distinct **impact cones** — interface (typed dependents), definitional (value-readers/unfolders), instance/search (affected candidate universes), syntax (GrammarEpoch descendants), executable (codegen/native roots), evidence (receipts and releases whose exact bytes changed), and the conservative potential cone through opaque edges — the same cones the Ledger's matrix consumes and reviewers read as release risk.

**15.2 Unification forensics.** `sound`-mode unification failures report the **minimal disagreement**: the smallest subterm pair where matching failed, the transparency setting in force, the approximation stage that gave up, and the assignment trail — rendered as a diff, not a wall. The Reference's "type mismatch" walls remain available verbatim in `faithful`.

**15.3 Instance forensics.** Synod's provenance rendered as the "why not" tree: every candidate, its failure reason, the priority order, the cycle/depth events — with a one-line head ("no `Foo Bar` because the only candidate needs `Baz Bar`, which has no instances") above the expandable tree. Both forensic surfaces are projections of one mechanism: the **semantic blame slice** — the minimal (or near-minimal, honestly labeled) causal subgraph sufficient to explain a failure, an instance choice, a slow build, or a surprising invalidation, with stable node ids so robot clients get the typed slice and humans get the rendering.

**15.3b Semantic diff & merge.** `fln diff` speaks levels, not lines: **source** (bytes/ranges) → **syntax** (lossless + expanded trees) → **interface** (names, universes, binders, types, attributes) → **proof** (proof-term identity and dependencies) → **reduction** (defeq/reducibility behavior) → **extension** (instances, simp sets, notation, env-extension deltas) → **executable** (IR/native products) → **evidence** (receipts) — so "proof-only change" is a checkable claim, not a reviewer's guess. On top of diff sits **semantic merge**: two branches merge automatically when the graph proves their transactions commute (disjoint declaration names or compatible additions; no grammar/extension write conflicts; no read-after-write between them; generated-name spaces stable) *and the combined declarations re-check through Crucible*; when text conflicts but semantics are independent, Palimpsest proposes the merge patch; when text merges cleanly but instance/simp/notation effects collide, it **blocks the silent merge and explains the semantic conflict** — the failure mode Git cannot even see. Merged results are replayed in both orders where order could matter, and nothing merges around the kernel.

**15.3c Fragility, advisorily.** Evidence-backed fragility signals per declaration — reliance on declaration/instance order ties, long or branching instance searches, near-miss simp rewrites in large sets, dependence on `Opaque`-classified extensions or private APIs, proof terms that change under harmless source perturbation, high rebuild centrality, repeated cross-epoch repairs, unusually large kernel normalization cost — each linked to concrete graph nodes, individually weightable and disableable, strictly advisory (D3), and constitutionally barred from influencing Crucible. `fln why-trusts <decl>` walks the trust cone (axioms, `sorryAx`, `native_decide` receipts and their tiers, `@[extern]`/plugin surfaces, `--trust` extensions) as a query over `CHECKED_BY`/`LOADS_LIBRARY` edges — the answer the ecosystem currently assembles by grep and folklore. Proof graphs can reveal proprietary structure, so exports are local-first with field-level redaction, pseudonymous ids, per-family export policy, and secret-taint propagation from the capability layer; a sanitized deterministic export exists for bug reports. Storage: the graph lives in-process on franken_networkx over frankensqlite-backed journals; an **optional export to frankengraphdb** (the suite's property-graph engine, as it lands) is the frontier lane for fleet-scale queries — derived, rebuildable, never load-bearing for soundness.

**15.4 Time travel.** Goal-state scrubbing through a proof (every tactic step's state addressable), elaboration replay of any declaration from its recorded closure (`fln replay <decl>` — a deterministic re-run, breakpointable), and **repro bundles**: one command packs a bug's input closure so every report is a replayable artifact, the manim plan's discipline applied to elaboration.

**15.5 Build forensics.** Because the Ledger journals into SQL, the questions teams actually ask are queries: slowest declarations this week, invalidation-cone sizes per commit, heartbeat consumers, instance-search hot spots, cache hit rates per module — `fln doctor --sql` and shipped report views (§20).

## 16. The leapfrog surfaces

**16.1 Bloodhound (fln-hound, frankensearch).** In-toolchain library search: lexical + semantic over statements, names, docstrings, and simp/instance membership, indexed incrementally at Ledger commit, queried from the editor (`$/lean/rpc` widget), the CLI, and Envoy. Serves `exact?`/`apply?`-class premise retrieval as a *candidate generator* whose candidates are verified by actual elaboration — ranking is UX, proofs are proofs. Loogle-style structural patterns ride the same index with a type-shape fingerprint lane. The charter is wider than search: Bloodhound is the **candidate-production system** — retrieval, bounded type-directed synthesis, proof repair, counterfactual analysis, and minimization — under one law: every candidate is untrusted text/terms until Athanor elaborates and Crucible accepts it. Results arrive as `CandidateCard`s (source/term, provenance and retrieval channels, required imports/instances, rank components, elaboration outcome, kernel outcome, fragility signals from §15.3c, replay identity) in three grades — **initial** (low-latency lexical/type-head/local-context channels), **refined** (structural unification, graph ranking over §15's cones, prior-use paths, optional learned rerank), **verified** (kernel-accepted under the exact snapshot) — and a failed refinement never withdraws a valid initial result. **Proof repair** consumes the causal graph when an edit or epoch migration breaks a previously-accepted proof: diff old/new environment and extension graphs, find the earliest failed constraint cell, re-elaborate the old proof with holes around the blame slice, search replacements (renames, binder-order shifts, bridge lemmas, instance-priority shifts, simp-set drift, termination adjustments, import moves), validate each minimal patch through the full pipeline, rank by semantic change size and fragility delta, and emit a patch transaction with rollback — the migration autopilot's engine (§13.6). **Counterfactual analysis** replays a verified proof under altered policy ("without this import?", "which simp rules are load-bearing?", "does a stricter capability profile block it?") on branch-local snapshots; **minimization** optimizes among verified terms along a declared objective vector (dependencies, term size, kernel cost, fragility, private-API use) with the trade-off shown, because "shortest" is not a universal good. Learned components (frankentorch tier) obey the standing rules: local-first, model/version identity on every card, deterministic inference profile where supported, candidates-only epistemics, and **removal leaves a complete deterministic lexical/type/graph baseline** — the feature degrades, never the trust story.

**16.2 Folio (fln-doc, franken_markdown).** doc-gen4-surface-compatible documentation as a Ledger facet: native HTML/PDF, fmd-math for `$…$`, Quill-fed highlighting, franken_networkx dependency and instance graphs, incremental (only changed declarations re-render). Mathlib's docs become minutes, local, and offline. Documents are **receipt-bound**: a rendered page embeds the logical root and receipt links for every checked excerpt, and a later source change marks the artifact *stale* rather than leaving a "verified" badge lying — literate proof documents whose claims carry evidence, in the Witness discipline of §20.4. The frontier lane is delightful and honest: franken_manim consumes Palimpsest event sequences to render **animated proof explanations** — goal transformations under a tactic, unification constraints resolving, induction structure, before/after of a repair — explanatory projections that link to checked declarations and are never themselves certificates.

**16.3 Envoy (fln-mcp, fastmcp_rust).** The agent door: MCP tools over Lantern's session API — `goal_state`, `apply_tactic` (against forked snapshots, budgeted), `search_lemmas` (Bloodhound), `check_term`, `eval` (capability-scoped), `get_certificate`, `ledger_query`, `trace_query` (Palimpsest). Budgets (heartbeats, wall, memory) are first-class parameters; every tool result carries provenance. Read tools are `sound`; write-to-file tools are explicit and default-off. The 2026 proof-search ecosystem currently builds this layer per-paper by scraping the human LSP; Envoy makes it the product. The session primitive is the **`ProofStateSnapshot`**: a versioned, schema-registered, canonical serialization of an elaboration point (goals with local contexts, metavariable assignments, environment logical root, journal position, option set) — round-trippable (a snapshot *is* a resumable state, not a description of one), stable across releases, structurally shared with the live store, and the format every tool below speaks. Authority is **macaroon-scoped** (asupersync's tokens): a credential names the environment roots it may read, the budget envelope (heartbeats, wall, memory, expanded-weight), the consensus policy it may request, and whether write-tools exist at all — attenuable delegation for multi-tenant fleets, with every check receipted. Robot responses carry schema/epoch/profile versions, request and snapshot ids, resource facts, data grade (provisional/verified), and evidence links; mutating tools carry idempotency keys and preconditions. On top rides **transactional multi-agent orchestration**: fork a branch per agent, assign disjoint impact regions from §15's cones, stream scoped context packs, accept *patch proposals* (never direct trusted mutations), re-elaborate and kernel-check each, detect semantic conflicts via §15.3b, merge deterministically, and emit a mission receipt with a rollback root — agent success measured by accepted kernel-checked commits, not by "done" messages. A remote **check service** completes the loop: submit a sealed capsule delta, receive full-policy consensus checking under budgets and a signed transparency-logged receipt — "check once anywhere, verify everywhere" as an operating capability (distribution details in §13.7's trust model: workers may lie, results validate locally, disagreement blocks and never votes).

**16.4 The terminal InfoView (frankentui).** `fln goals` — the live proof panel over SSH/tmux, where this ecosystem's agents and their operators actually live; widgets render as text/ANSI projections, with franken_node's JS host held as the exploratory path to full widget fidelity outside VS Code.

**16.4b The WASM Judge (fln-wasm).** K1 plus the independent checker compiled to WASM (§8.3b, §8.6): sealed-capsule verification in a browser tab — drag a `.flnpack`, watch the kernel re-check it against the same vectors with the same bits; receipt and transparency-log inclusion validation; declaration/dependency browsing over the embedded graph slice. Zero-install third-party verification is the public face of the trust story: the strongest possible demonstration that FrankenLean's trusted core needs nothing from a host — no filesystem, no network, no allocator tricks, no platform libm. Tier-1 scope is the Judge; a full in-browser elaborator is explicitly tier-2-if-ever (the daemon is the right answer for elaboration).

## 18. Tribunal: the verification apparatus (fln-conformance)

The Tribunal reorients the manim Gauntlet's stance for a domain where the oracle is executable: *the Reference runs inside the harness*, as the differential oracle, forever — and under the Oracle-Only Law the Tribunal is not QA but the second-largest subsystem in the program, because every fidelity claim the Mirror makes is proven here or not at all. When sources disagree, precedence is declared, in order: (1) the pinned epoch's **core theory** and the Judgment Specification — no Reference crash or bug can force acceptance of an invalid declaration; (2) **observed Reference binary behavior** on the matching platform; (3) Reference **sources and official tests** (explanatory, fixture-deriving); (4) **stable downstream dependence** (unchanged real projects demonstrating the de-facto contract); (5) **documented Franken divergence** (Behavior Notes); (6) **unclassified** — which blocks the claim rather than being guessed. Every fixture records its oracle kind, and a test may never quietly downgrade its comparison class to pass. The **epoch laboratory** hosting all of this — official archives with hashes, executable/symbol/header inventories, CLI and LSP transcripts, artifact fixtures, quirk/defect ledgers, environment-normalization recipes, exact reproduction commands, per platform — is immutable once published; discoveries create a reviewed, hashed revision.

**18.1 The Parity Ledger.** Row-per-symbol over the surfaces: language constructs, options, attributes, extern intrinsics, the Mirror's `Lean.*` façade rows, CLI flags, LSP methods, olean/ilean fields, ABI symbols, Lake verbs and facets. Columns: surface · symbol · kind · semantic status (**same** | **improved** (Behavior-Note link) | **tiered** | **excluded**) · **evidence level (L0–L4, §4.2)** · mode applicability · oracle kind · comparison class · fixtures · determinism class · claim state (`OBSERVED`/`TARGETED`/`HYPOTHESIS`/`PROVEN`/`BLOCKED`, §20.4) · freshness · notes. Rows are machine-readable and may not be summarized away: dashboards aggregate, but release gates reason over required rows and named exclusions. Regenerates the coverage dashboard; CI fails on regression; headline percentages are derived, never asserted.

**18.2 The test planes.**
- **Upstream conformance import.** The Reference's own test suite at the pin, run under FrankenLean with expected-outcome mapping; deviations triaged into the Ledger. Corpus families are named so coverage is auditable: **C0** micro-fixtures isolating single rules; **C1** official source-derived tests and stdlib modules (test data, never linked code — D8); **C2** ecosystem packages chosen for metaprogramming/FFI/Lake/syntax weight; **C3** artifact archaeology (valid + mutated oleans/ileans/C/objects/traces, every offset and tag targeted); **C4** native-ABI probes both directions; **C5** recorded LSP sessions; **C6** build/package variants incl. interrupted updates; **C7** adversarial/resource bombs; **C8** incremental/metamorphic edit sequences with expected cones; **C9** bootstrap/reproducibility (clean-machine, air-gapped, path/locale variation).
- **Differential elaboration (T2/T3).** Corpus files elaborated by both implementations; compared at tiers — T2: acceptance + diagnostics positions/severities + statement-level environment identity; T3: term-level identity up to registered normalization (alpha, mdata policy), with residual term divergences triaged (most are fidelity defects; a few become documented free choices with pinned CGSE policies). Comparison classes are explicit per row — exact bytes; normalized bytes under a *versioned* normalizer (normalizers are code with fixtures, may strip only declared-nonsemantic fields, and can never discard an error body to make a test pass); structured semantic equality; ordered or partial-order event equality; compile/link/run behavior — and the declaration differential additionally cross-checks both implementations' outputs through Crucible *and* `fln-checker`, with the standing nuance of §8.2: a Reference-accepted, Crucible-rejected declaration opens an oracle/spec investigation before any defect is assigned.

Traces are fixtures, not components; their generator lives behind D8's wall.

- **Codec rigs.** olean read/write round-trips (FL-INV-04), mixed-producer builds (Reference-built imports under FrankenLean and vice versa), corrupted-input fuzzing with resource budgets (a hostile olean is a DoS surface even in safe Rust).
- **ABI cross-load matrix.** Reference-built plugins ⇄ FrankenLean host, FrankenLean-built plugins ⇄ Reference host, per platform.
- **Protocol rigs.** Recorded editor sessions replayed against both servers; wire-level diffing with registered nondeterminism classes (request interleaving) normalized; vscode-lean4 smoke matrix in CI.

Mocked boundaries are fine for unit tests and rejected by the evidence gate.

- **Metamorphic laws.** Semantics-preserving transforms (comment/whitespace churn, independent-decl reordering, alpha-renaming of locals, universe renaming, hygiene-safe macro refactors) must preserve environments and — for the Ledger — invalidate nothing (the incremental law's own test).
- **Torture (asupersync lab).** The daemon and build fabric under virtual time with cancellation storms, fault injection, crash-recovery of the frankensqlite stores, and seed-replay of every failure; DPOR-bounded exploration of the scheduler's merge protocol.

**18.3 Determinism closure (certified).** The content-hashed input closure (sources, SUITE.lock, toolchain, options, plugin hashes, mode, CGSE policy epoch) ⇒ bit-identical environments, diagnostics, oleans, and manifests across the certified matrix (linux-x86-64, linux-aarch64, macos-aarch64; windows-x86-64 functional from W1, bit-certification a separate declared decision — the manim precedent). **18.4–18.7 (folded above into §18.2's rigs).**

**18.9 Shadow-run promotion.** Any lane replacing a conservative path (K2 optimizations, Anvil engines, Iron tiers, epoch bridges, Ledger precision upgrades) promotes through the matrix protocol of §12.6 — dual-run, diff, promote per cell, keep sampling, auto-demote on breach — and the promotion state is itself a queryable artifact.

**18.10 The lockstep dev harness (ORACLE_FALLBACK).** During construction only: a diagnosis mode runs the Reference beside FrankenLean on the same inputs inside the Tribunal's sandbox, compares products live, and can fetch an oracle artifact to bisect a divergence. ## 20. Observability & developer experience

Owned structured tracing (build → module → declaration → phase) with the Palimpsest event schema; per-run phase breakdowns (parse, macro, elab, instance search, kernel, VM, IO) so optimization begins by deleting work; `fln doctor` (environment, toolchain, closure, plugin, capability report — plus the derived execution plan and the D2 tools' presence); the build database as SQL (§15.5) with shipped views; frankentui progress and dashboards; repro bundles everywhere a failure surfaces; the public Parity Ledger dashboard and the T3 term-identity ratchet as first-class DX artifacts; provenance manifests on every artifact.

**20.4 Witness: the claim-to-proof matrix.** Every public claim — compatibility, soundness, determinism, performance, recovery, security — is a row: claim id · exact text or template · allowed evidence state (`OBSERVED` measured with a no-mock reproduction command | `TARGETED` gate specified, proof incomplete | `HYPOTHESIS` research annex | `PROVEN` a formal result with stated scope | `BLOCKED` | `RETIRED`) · owner workstream · scope (epoch/platform/mode/corpus) · required gate and artifact schemas · freshness predicate · forbidden stronger wording · fallback text. Documentation CI parses the docs against the matrix and rejects wording stronger than the evidence permits; coverage counts are generated from the ledgers, never hand-maintained. A benchmark is `OBSERVED`, never `PROVEN`; a Crucible-checked theorem is `PROVEN` relative to the assumptions its receipt names. Companion machinery: **safe mode** — on an `InternalFault`, checker disagreement, or corruption detection, the affected process fails closed, quarantines its caches, and requires independent revalidation before its outputs are trusted again; **auto-demotion** — a promoted lane (§18.9) that breaches its thresholds demotes itself and files the incident bundle. The system explains its own behavior with the same rigor it demands of proofs.

### 22.3 The convergence gates

**G1 — "The Independent Judge."** Marrow+Grimoire+Crucible check **every mathlib olean at the pin** with zero acceptance divergence against the Reference toolchain; PG-1 met; `fln check-olean` ships as a standalone auditor. The program's first public artifact is the strongest independent verification the Lean world has ever had — before FrankenLean has elaborated a single file.

**G2 — "The Golem Wakes."** User-class compiled Lean code executes on Golem with `#eval`/IO/Task parity on the fixture corpus; Reference-built plugins load and run (ABI door, inbound); initialization ordering proven; PG-7 interpreter floor met on early corpora; the intrinsic census fully populated with fixtures.

**G3 — "The Mirror Holds."** The native toolchain elaborates `Init` and `Std` end-to-end **from source** — every builtin either native behind its façade row or library-code source-elaborated, zero upstream implementation executing anywhere — with T2 identity against the Reference, the stdlib double-elaboration fixpoint bit-stable, and the metaprogram-corpus rig (a curated mathlib-tactic slice through the façade) green at its declared L-levels. The `ORACLE_FALLBACK` compile-out check passes on the gate build.

**G4 — "The Corpus Whole."** Full mathlib elaboration: T2 = 100%, T3 measured with a published ratchet, PG-2(G4) met, olean outputs consumed by the Reference toolchain in the mixed-build matrix. This is the gate that makes "an implementation of Lean" a fact rather than a claim.

**G5 — "The Daemon."** Lantern serves vscode-lean4 unmodified across the smoke matrix; PG-4/5/6/L met; the session API live; Verdict at PG-9; FrankenLean-built plugins load in the Reference (ABI door, outbound).

**G6 — "Sovereign & Beyond."** Iron builds real executables with no C compiler; Ledger GA with cache federation and PG-3/PG-10; the leapfrog surfaces shipped; W12 distribution complete; `--reproducible` certified across the matrix. The toolchain is one static binary set that installs in seconds and owes the outside world nothing.

### 22.4 Continuous obligations

The differential rigs, the Mirror conformance harness, the golden-trace replay rigs, the mutation campaigns, the thread-matrix determinism runs, the codec round-trips, and the Corpus tiers run from the moment each exists, forever; gates add obligations and never retire them.

### 22.5 The epoch policy

The Reference releases monthly; FrankenLean pins epochs deliberately. An epoch advance is a mechanical **ratchet**: (1) fetch and hash the official release; (2) regenerate every census on every platform; (3) run the existing corpus differentially and mine new behavior into fixtures + fresh instrumented-oracle traces; (4) machine-diff the public/Meta API, artifact, ABI, CLI, Lake, and LSP schemas; (5) regenerate façade stubs and open migration beads for every moved symbol; (6) close or classify every new divergence (unclassified blocks the epoch's claim); (7) run the unchanged downstream corpus under both epochs; (8) publish the capsule, evidence bundle, and migration report — then move the pin. Cadence target: a new stable upstream release reaches R3 within four weeks and R4 by the following cycle, with lag publicly visible on the dashboard. Epoch engines install side by side, selected by `lean-toolchain` exactly as users expect; old capsules stay runnable Supported-epoch count is a product decision (initial posture: current + previous stable); the Ledger's epoch bridges (§13.6) are the pressure-release valve that keeps the treadmill from owning the roadmap.

## Appendix A — Kernel judgment & reduction inventory (normative contract)

The complete rule set Crucible implements, frozen per epoch; each row carries fixtures and a Reference-source anchor. `KERNEL_CONTRACT.md` is W3's *first* deliverable — written before the engines, rule-anchored to Reference source lines, and the document all engines and the Tribunal's generators are built against (§8.1b). Summary of the inventory (full table generated at W3):

- **Typing:** bvar/fvar/sort/const/app/lam/forallE/letE/lit/mdata/proj rules; universe-parameter arity and validity; local-context discipline.
- **Levels:** normalization of zero/succ/max/imax/param; `imax _ 0 = 0`; level defeq under normalization.
- **Defeq:** beta; eta for functions; eta for structures; delta ordered by definitional height with reducibility-independence; zeta; iota (recursor rules incl. K-like subsingleton elimination); definitional proof irrelevance in `Prop`; `Quot.lift`/`Quot.mk` computation; `Nat`/`String` literal rules and the accelerated `Nat` operation set (add, sub, mul, div, mod, pow, gcd, beq, ble, blt, land/lor/lxor/shift family as the pin defines); lazy unfolding discipline and whnf caching semantics (K2) with K1 as the definitional reference.
- **Inductives:** strict positivity; universe constraints and large-elimination rules; mutual blocks; nested-to-mutual compilation; reflexive occurrences; recursor and `casesOn`-family generation obligations the kernel validates; structure eta/projection coherence.
- **Declarations:** axiom/def/theorem/opaque/quot/inductive admission rules; `partial`/`unsafe` marking and quarantine; environment uniqueness and hygiene rules.
- **Resources:** heartbeat and recursion-depth accounting hooks (§4.2) and the typed budget algebra (§8.2c), counted but never semantic in the mathematical sense — an exhaustion is `Inconclusive`, a verdict about a run, never about a term (FL-INV-07).
- **Admission machinery:** the typestate gate (`UnvalidatedExpr` → `ScopedExpr` → `CheckedExpr`), the `DeclarationEnvelope` schema, the four-way verdict, and the Judgment-object/consensus schemas of §8.3c.

## Appendix B — ABI & olean conformance contracts (extraction law)

`ABI_CONTRACT.md` and `OLEAN_CONTRACT.md` are **generated artifacts**: checked-in extraction scripts parse the pinned Reference sources (`lean.h`, the runtime's layout definitions, the olean writer/loader) and emit (a) human-readable contracts with per-field provenance, and (b) the Rust constants modules Marrow and Grimoire compile against. CI re-extracts on every run of the contract job and fails on any drift among pin, contract, and constants. Content inventory: object header packing; per-category tag values; constructor scalar-packing and field-offset rules; boxing rules per builtin type; the persistent/shared/single RC semantics; the exported `lean_*` function census with signatures and ownership conventions (owned/borrowed per parameter); olean header fields and versioning; region relocation-table format; `ModuleData` field inventory and extension-entry encoding. The round-trip law (FL-INV-04) and the mixed-producer matrix are the contracts' executable form. The extraction additionally emits: the `.ilean` field contract (§7.3b); the exported-symbol status taxonomy (§6.5); per-symbol ownership conventions; and the semantic-hash/byte-hash schema registry (§7.3) so every durable format is specified once and generated into both the codecs and the conformance corpus.

## Appendix C — The builtin census & the Mirror partition (method)

At each epoch, extraction walks the pinned `Init`/`Std`/`Lean` sources and emits the **builtin census** over the entire reachable declaration surface: symbol, Lean signature, universe/binder telescope, module, kind, attributes, `@[extern]`/C symbol where present, effect class, and — the Mirror's load-bearing column — the **partition class** of §4.3: `toolchain-api` (native implementation required; façade stub generated; equivalence class + fixtures + L-level per row), `library-code` (source-elaborated by our toolchain; fixture = elaboration identity), or `user-facing-data` (types/structures whose layout is contract). Golem's intrinsic table is generated from the `@[extern]` subset; the façade registry (fln-elab) is generated from the `toolchain-api` subset; behavioral fixtures come from the official tests, the metaprogram corpus, and the instrumented oracle's traces.
